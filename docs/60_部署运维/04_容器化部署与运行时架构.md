# Part A · 架构说明

## 1. 一张图：四层 + 六条数据流

```
                    ┌─────────────────────────────┐
          ①登录/指令 │   前端工作界面 User Workspace │ ⑤异步通知(触发UI重绘)
        ┌──────────▶│   登录·项目选择·作业入口     │◀──────────┐
        │           │   事件驱动:收通知→拉文件刷新 │           │
        │           └──────────────┬──────────────┘           │
        │                          │⑥API直连取文件            │
        │           ┌──────────────▼──────────────┐           │
        │           │     Manager 协调后端         │           │
        │           │  鉴权·容器调度·上下文准备     │           │
        │           └──────┬───────────────┬──────┘           │
        │        ②鉴权/拉项目│               │③拉起容器+指配目录 │
        │           ┌──────▼──────┐  ┌─────▼───────────────┐  │
        └───────────│  数据中心    │◀▶│   Claw 容器(沙箱)     │──┘
              ⑥API   │ Postgres·项目│④ │ LangGraph 执行引擎    │
                     │ ·Skill·会话  │零 │ 结果写映射目录·通知前端│
                     └─────────────┘拷贝└─────────────────────┘
```

| # | 数据流 | 含义 | 图例色 |
|---|--------|------|--------|
| ① | 前端→Manager | 登录 / 业务指令 | 请求指令流 |
| ② | Manager→数据中心 | 查密码鉴权 / 拉取归属项目 | 调度控制流 |
| ③ | Manager→Claw | 分配拉起独立容器 + 指配映射目录路径 | 调度控制流 |
| ④ | 数据中心↔Claw | **数据零拷贝物理映射**（双向） | 物理目录映射（虚线） |
| ⑤ | Claw→前端 | 异步通知消息（触发 UI 重绘/拉取） | 事件通知流 |
| ⑥ | 前端→数据中心 | API 接口直连，获取项目文件数据 | API 数据流 |

> 一句话：**Manager 调度、数据中心存数据、Claw 容器跑 LangGraph、前端事件驱动刷新；作业产物走"零拷贝目录映射"而非 HTTP 上传。**

## 2. 四层职责详解

### 2.1 前端工作界面（User Workspace UI）
- 用户登录与项目选择、作业执行入口。
- **事件驱动**：自身不轮询；收到 Claw 的通知后，走 ⑥ API 从数据中心拉最新文件刷新界面。
- 渲染模型：**`render(state.json)` 纯函数**（数据中心 README §state.json）。

### 2.2 Manager（Core Coordinator Backend）
- **鉴权**：到数据中心查密码鉴权，成功后拉取用户归属项目交给前端。
- **容器调度**：为每个用户单独拉起专属 Claw 容器实例（多租户隔离）。
- **上下文准备**：拉容器时把数据中心对应目录（Skill / 项目数据 / 会话）bind-mount 进容器。

### 2.3 数据中心（Data Center & Storage）
- **核心原则：数据 ≠ 代码**——只放「数据 + 资产 + 技能定义」，**不放可执行代码**（step 逻辑属于 Claw 应用层）。
- 内置 **Postgres**（用户 / 角色）+ 文件存储（项目数据 / Skill / 模型 / 本体 / 会话）+ 对外 **API**。
- 三级隔离布局（权威结构，见 [`aida-datacenter/README.md`](../../../aida-datacenter/README.md)）：

```
aida-datacenter/
├── skills/<skill>/                    # 技能定义层(全局共享)
│   ├── SKILL.md  module.json
│   ├── dashboard.template.json        # UI 初始态模板(render 起点)
│   └── assets/                        # 固定资产(评估底表/风险库/报告模板)
└── projects/<projectId>/skills/<skill>/runs/<runId>/   # 三级隔离·根除并发覆盖
    ├── input/  runtime/  output/
    └── state.json                     # ★ 该 run 唯一真相(读契约)
```

### 2.4 Claw 容器（Isolated Execution Sandbox）
- **后台实际作业 = 真正的执行引擎**：即 `aida/agent` 的 FastAPI(7401) + LangGraph（02 范式的落地体）。
- **目录写映射（零拷贝）**：`SkillContext.work_root` 指向某个 `runs/<runId>/`，作业结果直接写 `output/` + `state.json`，数据中心即时可见，无需上传。
- **通知前端刷新**：作业状态变更时异步通知前端。

## 4. 两层持久化澄清（避免与 02 矛盾）

02 范式写「持久化用 SqliteSaver，**不引 Postgres**」，本图却有 Postgres——**不矛盾，是两个层次**：

| 层 | 用途 | 选型 | 位置 |
|----|------|------|------|
| **平台层持久化** | 用户 / 角色 / 鉴权 / 项目索引 / 会话目录 | **Postgres** | 数据中心（多租户共享） |
| **运行态持久化** | 单次 run 的 checkpointer（中断重启不丢） | **SqliteSaver** | 每个 Claw 容器内 |

> 开发者最易困惑点：**Postgres 管"谁能跑、跑哪个项目"，Sqlite 管"这次 run 跑到哪一步"。** 02 说的"不引 Postgres"特指后者（run 状态），仍成立。

## 5. 已知矛盾与演进（诚实标注）

1. **前端更新机制有两种说法（必须收敛）**：
   - 「Claw 对比画板」讲 **SDUI node patch ops**（`{op:merge,target,value}`）。
   - 数据中心 README 审计 B3 明确选 **推送 state.json 增量**，*不是* SDUI node patch。
   - **以数据中心 README 为准**：实时更新 = state.json 增量；SDUI 是"渲染器复用"层面（同一渲染器换 schema），不等于"用 node patch 做实时更新"。两者别混。
2. **agent 代码多库副本**：`aida/agent`（我实测跑的）、README 提到的 `aida-vite/agent`、`claw-delivery-ui/agent` 存在多份副本且内容有别。**部署前必须收敛为单一真相**，否则"改 A 跑 B"。（这正是本轮测试踩过的坑。）
3. **工作区结构演进**：旧单例 `zhgk-desktop/{Start,Input,Output}`（当前实测用）→ 新三级隔离 `projects/<pid>/.../runs/<runId>/`。新模块**一律用新结构**，勿沿用单例。
4. **Manager 参考**：当前体系无 Manager；`hiclaw/manager`（旧项目）可作设计参考起点（未审，仅线索）。

# Part B · 开发者对接操作地图

> 02/03 已讲「Skill 代码怎么写」（A+B 双层、BaseSkill/BaseStep、工具/守门）。本部分只讲**增量**：写好的模块如何接入这套四层部署、数据放哪、前后端怎么联通。

## 6. 新业务场景 Skill接入四层 —— 五步

```
第 1 步 · 数据中心建技能定义（不写代码）
  aida-datacenter/skills/<skill>/
    ├── SKILL.md                 ← A 层门面(与 02 §3.1 一致)
    ├── module.json              ← 模块元数据(flow / HITL / task 列表)
    ├── dashboard.template.json  ← 前端 render 的初始态(UI schema 起点)
    └── assets/                  ← 固定资产(底表/模板/规则库)

第 2 步 · Claw 写执行引擎（B 层代码,在 agent/skills/<skill>/)
  按 02 §3.4 / 03 §7：BaseSkill + steps；LLM 走 ctx.invoke_llm；工具走 ctx.call_tool
  ★ work_root 不写死——由注入决定,指向 runs/<runId>/(见第 4 步)

第 3 步 · 冻结读契约 state.json
  约定本 skill 的 state.json 形状(metrics / outputs / steps / alerts...)
  前端 render(state.json) 与后端投影都以它为唯一真相；带 schemaVersion
  ★ 产物清单以 state.outputs 为准,不靠扫 output/ 目录(跨 run 残留)

第 4 步 · 接目录映射（Manager 上下文准备的契约面）
  后端 SkillContext.work_root = projects/<pid>/skills/<skill>/runs/<runId>/
  input/ 收用户上传 · runtime/ 中间态 · output/ 产物 · state.json 真相
  单机期:本地路径注入即可；容器期:Manager bind-mount 该目录进容器

第 5 步 · 接前端更新
  单机期:复用 SSE 推 state.json 增量,前端 render 刷新
  多租户期:改由 Manager/通知网关转发 Claw 事件 → 前端拉数据中心 API
```

## 7. 目录映射契约（对接的硬接口）

| 角色 | 读 / 写 | 路径 | 约束 |
|------|--------|------|------|
| Claw 后端 | 读写 | `runs/<runId>/{input,runtime,output}` + `state.json` | work_root 注入，**不跨 run 访问** |
| 数据中心 | 存储 | `projects/<pid>/skills/<skill>/runs/<runId>/` | 三级隔离，租户/项目/run 不碰撞 |
| 前端 | 只读 | 经 ⑥ API 读 `state.json` + `output/` | 清单以 `state.outputs` 为准 |
| Manager | 装配 | 拉容器时 mount 上述目录 + `skills/<skill>/`（只读资产） | 容器只见自己租户的目录 |

## 8. 分阶段落地路线（务实）

| 阶段 | 形态 | 已有 | 要补 |
|------|------|------|------|
| **P0 单机样板**（现在） | 单进程 agent + 本地工作区 | ✅ zhgk 跑通 | 收敛 agent 多副本、迁到三级隔离目录 |
| **P1 数据中心成型** | 独立数据中心 API + Postgres | ⚠️ 目录已有 | 建 Postgres(用户/角色) + 对外只读 API |
| **P2 Manager + 容器化** | Manager 调度一用户一容器 | ❌ | 建 Manager(鉴权/编排) + Claw 镜像化 + bind-mount |
| **P3 多租户事件** | 跨容器通知网关 | ❌ | SSE→网关/WS，Claw 事件经 Manager 转前端 |

## 9. 对接清单（Checklist）

```
□ 技能定义  aida-datacenter/skills/<skill>/ 建 SKILL.md + module.json + dashboard.template.json + assets/
□ 执行引擎  agent/skills/<skill>/ 按 02/03 范式写 BaseSkill + steps（守门 lint 通过）
□ 读契约    冻结 state.json 形状 + schemaVersion；产物以 state.outputs 为准
□ work_root 指向 runs/<runId>/，不写死、不跨 run
□ 隔离      用 projects/<pid>/.../runs/<runId>/ 三级结构,勿用旧单例 Start/Input/Output
□ 前端      render(state.json)；实时更新走 state.json 增量(不是 SDUI node patch)
□ 持久化    run 状态 SqliteSaver(容器内)；用户/项目 Postgres(数据中心)——分清两层
□ 通知      单机 SSE；多租户预留"事件经 Manager 转发"接口,勿把前端地址写死进 Claw
```

## 10. 一句话总结

**业务场景 Skill = 数据中心放「技能定义 + 隔离运行数据」，Claw 容器跑「LangGraph 执行引擎」，Manager 管「鉴权 + 一用户一容器 + 目录映射」，前端「render(state.json) + 事件驱动刷新」；产物走零拷贝目录映射、实时更新走 state.json 增量。当前竖向能力链已通（单机），横向多租户平台层（Manager / Postgres / 容器化 / 跨容器通知）待建——这就是从样板到平台的路线图。**
