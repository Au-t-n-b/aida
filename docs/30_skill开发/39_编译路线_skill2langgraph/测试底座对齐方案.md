# 测试底座对齐方案 · 最终形态（以 AIDA 本体为测试底座）

> 状态：Draft v1（2026-06-08）
> 适用对象：skill2langgraph 流水线开发者；AIDA Agent / Manager / 数据中心 / CI 各模块开发者
> 上游参考：`docs/30_skill开发/31_手写规范/AIDA-RUNTIME-CONTRACT.md`、`reference/design-prior/skill2langgraph设计.md（历史稿·未随本集导入）`（被本方案取代的 Harness 设计）、AIDA 仓库 `aida/docs/04_容器化部署与运行时架构.md`、`aida/AGENTS.md`
> 配合需求：见 `docs/skill2langgraph/requirements/测试底座/`（逐模块需求文档，本文负责"是什么/为什么/怎么验"，各需求文档负责"对应模块要交付什么"）

---

## 0. 一句话定位

把"生成出来的 Graph 是否正确"这件事的验证底座，从**一套需要持续对齐的 AIDA 复刻品（Harness）**，换成 **AIDA 本体自身**——以容器化的、用镜像 digest 钉死版本的真实 AIDA 作为测试沙箱。**删除"第二套实现"，对齐问题随之消失。**

---

## 1. 背景：为什么放弃 Harness

`reference/design-prior/skill2langgraph设计.md（历史稿·未随本集导入）` 设计了一个 "AIDA-compatible Test Harness"——"AIDA 执行底座的测试化版本"。它要逐面对齐 AIDA 的 6 个接缝（state / provider / 工具运行时 / HITL / 建图 / checkpoint）。

但 AIDA **自身仍在剧烈开发**（`04_容器化部署与运行时架构.md` §3 多处标 ⚠️/❌），是个**移动靶**。维护 Harness 等于维护一套追着 AIDA 跑的复刻实现，漂移**永远滞后于发现**（经典"测试全绿、上线全红"）。

**根本解法不是更努力地对齐，而是删掉被对齐的那一方。** 既然 AIDA 的执行引擎（`aida/agent`：FastAPI + LangGraph）本身可运行、且设计上就是"一用户一容器"的隔离沙箱，那就**直接拿它当测试底座**——"测试通过" 字面上就等于 "在真实 AIDA 上通过"，无第二套实现可漂移。

> 本方案**取代** prior 设计中的 "AIDA-compatible Test Harness" 一节作为对齐手段。Harness 里关于"三类验证 / 执行证据字段"的思想仍复用，但**证据来源从自造对象改为读 AIDA 原生产物**（见 §6）。

---

## 2. 最终形态架构

最终形态 = AIDA 的 P2/P3（Manager 调度 + 一用户一容器 + Postgres 鉴权 + 镜像化）。测试复用同一套基建，只是身份是"测试租户"、生命周期是"用完即弃"。

```text
                          skill2langgraph 流水线（CI 侧）
   ┌───────────────────────────────────────────────────────────────────────┐
   │  生成物：skill 模块（SKILL.md/module.json + BaseSkill+steps + state.json schema）│
   │       │                                                                  │
   │       ▼  ① 请求一次性测试容器(指定镜像 digest)                            │
   │  ┌──────────┐  ②拉容器+注入skill+bind-mount work_root   ┌──────────────┐ │
   │  │ Thin Test │ ───────────────────────────────────────▶│  Manager      │ │
   │  │  Driver   │                                          │ (测试租户鉴权) │ │
   │  │ (我方)    │ ◀── endpoint + runId ───────────────────│               │ │
   │  └────┬─────┘                                           └──────┬───────┘ │
   │       │ ③ HTTP 驱动 /agent/<skill>/start · /resume            │③拉起    │
   │       ▼                                                        ▼          │
   │  ┌─────────────────────────────────────────────────────────────────────┐│
   │  │  一次性 AIDA 测试容器  ==  生产同一镜像 digest                          ││
   │  │  · LangGraph 执行引擎（真身，非复刻）                                   ││
   │  │  · provider → ZHIPU_BASE_URL 指向【回放服务】(确定性，不烧 token)       ││
   │  │  · 外发工具强制 dry-run；外部工具端点→回放                              ││
   │  │  · work_root＝容器内一次性目录(run 态留容器)；输入/产物经数据中心 files API ││
   │  └─────────────────────────────────────────────────────────────────────┘│
   │       │ ④ 跑完产出真实证据                                                 │
   │       ▼                                                                    │
   │  state.json(唯一真相) + SqliteSaver checkpoint + Langfuse trace            │
   │       │ ⑤ 读出证据                                                         │
   │       ▼                                                                    │
   │  三类验证：行为 / 契约 / 运行稳定性 → 最终门禁 → 交付包                      │
   └───────────────────────────────────────────────────────────────────────┘
```

关键点：**测试容器与生产容器是同一个镜像 digest**；唯一差异是（a）provider 指向回放服务、（b）身份是测试租户、（c）run 用完即销毁。其余一律真身。

### 2.1 存储与概念分工（避免三个易混系统混淆）

| 系统 | 本质 | 存什么 | 谁用 |
|---|---|---|---|
| **数据中心** | 运行时**文件存储 / 项目云空间** | 项目文件/资产、项目、用户 | skill 运行时经 files API 读输入/写产物 |
| **文档库** | AIDA 各模块**文档孪生** | 设计文档 + 结构化能力上下文（含数据中心 API 的结构化镜像） | 开发者对齐 + 喂开发 agent；前门 S6 接地 |
| **Toolbox** | **工具运行时 + 注册中心** | 工具实现/版本 | skill 调用工具 / 复用查询 |

**存储三层落点（关键，决定各模块需求边界）：**

| 存什么 | 落点 |
|---|---|
| run 执行态（`state.json` / checkpoint / node 轨迹） | Claw **容器内** SqliteSaver |
| skill 定义 + 代码 + 新工具 | AIDA **镜像**（CI 烤入） |
| 项目文件（输入资料 / 最终产物） | **数据中心**项目云空间（files API） |

→ **数据中心不懂 run/skill，run 态不落数据中心。** 据此：run 态取证从容器读（§6）、skill 与新工具随镜像注入（§3）、项目文件经数据中心 files API 读写（数据中心 §R2）。

### 2.2 SDUI 边界（呈现投影，固定组件库）

SDUI 是 AIDA 的**呈现投影层**，**不是工具、不存数据中心**：skill 的 `sdui.py::project(state)→dict`（纯函数）把运行态投影成节点树，SSE 层每次 state diff 调用它并**流式推前端**，前端 `SduiNodeView` 通用递归渲染。

| 层 | 是什么 | 归属 |
|---|---|---|
| 组件库（节点词汇表 + 渲染器 + projector_base） | 平台级 UI 协议，**固定、专人维护** | AIDA 前端，随镜像/前端 digest；`lint_sdui_contract` 三方守门；**我方只消费** |
| skill 投影器 `project` | 把本 skill state 投影成树 | 生成物（**代码**）→ 镜像，随 skill；**只拼现有组件** |
| 运行时组件树 `sdui_doc` | 每次 diff 投影、SSE 流 | 运行态，派生、**不存储** |
| `dashboard.template.json` | render 初始态 | skill 定义 |

**复用-only（对照 toolbox）**：工具缺可"自实现 + 注册"；SDUI 组件缺**不能自造**（渲染器固定、三方一致），只能复用现有或**报缺口给维护者**（见 `docs/skill2langgraph/requirements/SDUI组件库需求.md`）。多数业务看板靠现有组件（stepper / KPI / 表格 / 产物栏 / HITL）即可。

- **编译期**：据「可用组件清单契约」（SDUI 维护者，R1）生成投影器；通用段（header/stepper/进度/产物/HITL）自动从标准 `state.json` schema 复用 `projector_base`；业务段（KPI/风险表/摘要）从 Raw Skill 的 §输出/§风险/§状态字段派生；**过 `lint_sdui_contract` 方可注入**。
- **前门**：Raw Skill **不设计 UI**，呈现从 state 自动投影、风格走设计偏好（领导汇报视角）；最多可选一个业务高度的"呈现要点"。

## 3. Thin Test Driver（薄驱动，我方拥有）

Harness 从"复刻 AIDA 执行底座"塌缩成一层薄壳，只干五件事，**不重写任何 AIDA 行为**：

| 步 | 动作 | 依赖模块（见需求文档） |
|---|---|---|
| 1 注入 | 把生成的 skill + 新工具**随镜像/加载机制装进容器**并注册（skill 定义+代码归镜像，非数据中心） | AIDA Agent §R3/§R10、Manager §R2 |
| 2 隔离 | 容器启动前 `ZHIPU_BASE_URL`→回放服务；外部工具端点→回放；强制 dry-run | AIDA Agent §R2/§R9/§R10 |
| 3 驱动 | HTTP 打 `/agent/<skill>/start`，按场景需要 `/resume`（测 HITL） | AIDA Agent §R7 |
| 4 取证 | 从容器读 `state.json` + checkpoint（run 态在容器内）+ Langfuse | AIDA Agent §R5/§R6 |
| 5 断言 | 喂真实证据跑行为/契约/稳定性三类验证 | 我方 |

Driver 自身**不需要其他模块开发**——它只是个 CI 里的客户端。其余四步全部依赖被测 AIDA 暴露稳定接缝，那些才是要写进需求文档的部分。

## 4. 版本钉死：镜像 digest = 契约版本

这是"始终对齐"的最终收口。不再维护语义化契约版本协商，而是：

> **对齐 ≡ 「测试用镜像 digest」 == 「用户生产容器镜像 digest」。**

- 测试从某个 AIDA 镜像 digest 起一次性容器跑验证。
- 生产用户容器跑**同一 digest** → 执行底座逐字节相同。
- 每个产物（Graph / 验证报告）记录"基于哪个 digest 验证通过"。
- AIDA 发新 digest → 流水线**无头批量重跑**所有已部署 Graph 的验证 → 仅把"仍绿"的提升到新 digest（见 §8）。

镜像 digest 直接当契约版本，比 semver 协商更硬：**不对齐移动靶，而是把移动靶钉成一串不可变快照。** 依赖 CI 镜像模块（见需求文档 04）。

## 5. 确定性策略与 toolbox 边界

真身底座的非确定源**不是只有 LLM**，而是 **LLM + 一切跨出容器边界的工具调用**。按真实 I/O 行为给 toolbox 划清边界，全部不引入第二套实现。

### 5.1 LLM 回放

1. **Provider 回放**：`get_llm()` 从 env 读 `ZHIPU_BASE_URL`（`aida/agent/llm.py:31`）。测试把它指向**回放服务**——一个 OpenAI 兼容端点，按 fixture 重放录制好的响应。需 AIDA 提供干净的 provider 注入接缝（含单例 temperature 固定问题，见需求文档 01 §R2）。
2. **录制档 vs live 档**：AIDA 已有 `eval_zhgk.py --fixture`（离线确定）与 live 两档先例。沿用：首次对一个新场景跑 live 录制 fixture，此后回放。

### 5.2 工具运行期边界（按 I/O 行为分类）

按工具的**真实 I/O 行为类别**划边界（行为类别是稳定的；具体工具集是临时的——见下方注）：

| 行为类别 | 现状示例工具 | 真实 I/O | 测试底座待遇 | 依赖 |
|---|---|---|---|---|
| 本地文件 | `read_file`、`doc_read_xlsx`、`doc_write_docx` | 读写 `work_root` 下文件 | **真跑**：输入由 `input/` 种子 fixture 提供（确定），输出写 `output/`，用完即弃 | 数据中心 §R2、Manager §R2 |
| 外发副作用 | `send_mail`、`send_welink` | 网络外发，env 门控 | **强制 dry-run** + 测试租户无真实收件人（双保险） | AIDA Agent §R9、数据中心 §R1 |
| HITL 拦截 | `present_choices` | chat_engine 拦截，不真执行 | 测试驱动**脚本化应答**（等价 resume 注入选择） | AIDA Agent §R7 |
| **外部网络** | 数据中心 API 类工具、MCP 类 | **真实 HTTP**（`urllib.request`） | **必须回放**——与 LLM 同等对待，按 fixture 重放 | AIDA Agent §R10、CI §R5 |

> ★关键：本地文件工具因 `work_root` 种子化而确定、外发工具因 dry-run 而无副作用，但**外部 HTTP/数据中心 API 调用必须像 LLM 一样走回放**，否则测试不可复现。这是工具回放接缝（AIDA Agent §R10 + CI §R5 回放服务扩到工具维度）存在的理由。
>
> **注（工具集是临时的）**：现 toolbox 模块**尚未开发**，工具暂放 `aida/agent/tools/` 由 AIDA 直接调用；`run_survey`、`zhgk_bridge` 等 skill-as-tool 是**妥协产物，不纳入最终设计**。本表按**行为类别**而非具体工具划边界，故 toolbox 成型、工具集变化后仍成立。toolbox 模块需求见 `docs/skill2langgraph/requirements/Toolbox模块需求.md`。
>
> **回放落地定为「端点重定向」**：工具的外部端点（如 `mcp_adapter.base_url`）必须可配置，测试时重定向到回放服务——与 `ZHIPU_BASE_URL` 同构。**不采**"容器内 fake tool runtime"，因为那等于引入第二套工具实现，与"用 AIDA 真身测试"的初衷冲突。端点重定向保持工具代码逐字节真身，只换它指向的地址。

### 5.3 工具解析策略与注入期边界

工作流对每个工具能力需求按三步解析（详见 `docs/skill2langgraph/requirements/Toolbox模块需求.md` §2）：

1. **复用**：优先复用 toolbox 已有工具（经文档库能力查询命中）。
2. **自实现**：toolbox 没有 → 工作流**自行实现新工具**，作为生成物**随 skill 一起打包**。
3. **注册回流**：全流程（含本测试底座三类验证 + `lint_tools`）通过后，把新工具**注册回 toolbox**（status `planned→stable`），供未来复用。

测试底座由此承担的职责：

- 注入时**既装 skill 也装随包的新工具**（不是只装 skill）。
- **断言**：**复用类工具**存在于该 digest 镜像/toolbox；**新工具**随包可加载且通过 `lint_tools`（AIDA Agent §R10）。缺失复用工具或新工具不合规 → fail-fast，不起 run。
- 新工具**注册回 toolbox 发生在验收通过之后**（post-run），**不是**测试前的拦截。

> 与旧表述的差异：先前写"缺工具即 `capability_gap` 拦下、不进测试"是错的。默认路径是**自实现 + 随包测试 + 通过后注册**；仅当能力连自实现都不可能（需不存在的外部系统）才退回 `capability_gap` 工单（文档库 §7）。

### 5.4 副作用与数据隔离总结

三层叠加：`send_mail`/`send_welink` 默认 dry-run、`present_choices` 被拦截（契约 §2.4）；测试租户**无真实联系人配置**（数据中心 §R1）；run 目录用完即弃。发不出真实通知、改不了真实数据、外部读全部走回放。

## 6. 取证与三类验证

**不再自造 `TestRunEvidence` 对象**，改读 AIDA 原生产物：

| 证据 | 来源（均为**容器内** run 态，非数据中心） | 出处 |
|---|---|---|
| 最终状态 / 产物清单 / steps / metrics / warnings | `state.json`（该 run 唯一真相，以 `state.outputs` 为准；run 态在容器内） | 04 §2.1、契约 §3 |
| node 轨迹 / state diff / checkpoint / resume | 容器内 SqliteSaver | 契约 §5、04 §4 |
| model_calls / tool_calls / latency | Langfuse trace（按 skill/step/run_id 聚合） | 契约 §1.2/§2.1 |

> 注：`state.json`/checkpoint 属 **run 执行态、落在容器内**（§2.1 存储分工），取证从容器读，不经数据中心。skill 最终**产物文件**才经数据中心 files API 回传项目云空间。state.json 是否同步给前端可见，取决于 work_root↔REST 桥接（§9 B12），与取证无关。

三类验证（沿用 prior 设计 §三类验证，喂真实证据）：

- **行为验证**：Graph 是否保留原始 Skill 语义（黑盒行为测试 × 真实证据）。
- **契约验证**：Graph 是否忠实实现 SkillIR。
- **运行稳定性验证**：在**真实 AIDA**里是否稳定——Graph 跑完或进预期 HITL、state diff 合 schema、provider 调用成功且输出可解析、工具成功（非 `Error:` 前缀）、HITL 用软中断表达、checkpoint/resume 满足要求、**SDUI 投影合规**（SSE `sdui` 事件纳入证据，用前端 `sduiCompliance` / `lint_sdui_contract` 校验投影只用现有组件且 props 合法）。

最终门禁：`accepted = behavior.passed ∧ contract.passed ∧ runtime.passed`。因为底座是真身，runtime 验证从"验证复刻是否像 AIDA"升级为"验证就在 AIDA 上跑过"。

## 7. 测试身份

最终形态有鉴权（Manager + Postgres），测试必须有身份，但**不是一个共享登录号**：

```text
专用测试 service tenant（测试用户，独立于任何真实客户，无真实联系人）
   └── 预置已审批测试项目 → 每次 CI run 用独立项目/目录隔离产物（run 态在容器内）
```

- 身份层：提供"谁在测 / 资源算谁的 / 能碰哪些项目"的鉴权与归账边界（数据中心人员管理 + 项目管理）。
- 隔离层：并发测试**按测试项目/目录隔离**云空间产物（数据中心无 run 概念；run 执行态在容器内 SqliteSaver）。
- 测试用户 + 种子项目随 digest 钉死、完全可复现（数据中心 §R1、CI §R2）。
- **严禁复用任何真实客户账户测试。**

> P0 回退说明（当前单机、无鉴权）：现阶段 AIDA 无 Manager/Postgres，端点开放。此时无需账户——直接对 `aida/agent` 进程注入 skill、provider 回放、强制 dry-run、一次性 work_root 即可。最终形态的租户身份是 P2 引入鉴权后才生效的层。

## 8. 再验证流程（应对 AIDA 演进）

AIDA 发布新镜像 digest 时：

1. CI 检出"当前生产 digest → 新 digest"。
2. 对所有已部署 Graph，用新 digest 容器**无头批量重跑** §6 三类验证（Graph 是可再生成产物，重验近乎零人力）。
3. 产出"仍绿 / 转红"报告：仍绿的 Graph 提升其"已验证 digest"；转红的回流编译管线重新生成或挂起。

这把"持续对齐一个移动靶"变成"对一串不可变快照做幂等的批量重验"。依赖 CI 镜像模块（需求文档 04 §R4）。

## 9. 前置阻塞与跨模块依赖总表

| # | 阻塞/依赖 | 责任模块 | 需求文档 | 不解决的后果 |
|---|---|---|---|---|
| B1 | `agent` 多副本收敛为单一真相 | AIDA Agent | 01 §R1 | "测的 AIDA" ≠ "跑的 AIDA"，对齐从根上破裂（头号阻塞） |
