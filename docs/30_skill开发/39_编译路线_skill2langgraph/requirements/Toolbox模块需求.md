# Toolbox 模块配合需求

> 状态：Draft v1（2026-06-08）
> 提出方：skill2langgraph 流水线
> 面向：Toolbox 模块开发人员（工具运行时 + 注册中心）
> 关联：`docs/skill2langgraph/requirements/文档库需求文档.md`（能力目录/读侧，与本文配对）、`docs/30_skill开发/31_手写规范/AIDA-RUNTIME-CONTRACT.md`（Tool 基类契约 §2.3）、`docs/skill2langgraph/测试底座对齐方案.md`（测试与注册前闸门）

## 0. 背景与现状

**Toolbox 模块当前完全未开发。** 工具暂时直接放在 AIDA 的 `aida/agent/tools/` 里由 AIDA 直接调用——这是过渡形态，不是最终架构。其中 `run_survey`、`zhgk_bridge` 等是当前的**妥协产物**（skill-as-tool 临时桥接），**不纳入最终设计**。本文描述最终形态下 Toolbox 模块应提供的能力，供该模块开发人员参考。

## 1. 边界澄清：文档库 vs Toolbox

二者相关但不同，必须分清，否则会重复造或留空白：

| | 文档库（doc library） | Toolbox（本文） |
|---|---|---|
| 角色 | 能力**目录 / 索引**（元数据、描述） | 工具**运行时 + 注册中心**（实现本体） |
| 读/写 | 主要**读**（grounding、复用查询） | **读+写**（加载工具、注册新工具） |
| 内容 | "有哪些能力、对应哪个工具、能不能做" | 工具的可加载实体、版本、注册接口 |
| 需求文档 | `文档库需求文档.md` | 本文 |

配对关系：**复用查询走文档库目录；工具实体的加载与新工具的注册写回走 Toolbox。** 注册一个新工具时，Toolbox 同步把文档库目录里该能力的 `status` 由 `planned` 更新为 `stable`（闭合文档库 §7 缺口回流环）。

## 2. 工作流的工具解析策略（skill2langgraph 侧，供 Toolbox 对齐）

编译管线对每个工具能力需求按三步解析：

1. **复用**：优先复用 Toolbox 中已有工具（经文档库能力查询命中）。
2. **自实现**：Toolbox 没有 → 工作流**自行实现新工具**，作为生成物**随 skill 一起打包**，在测试底座中与 skill 一同运行验证。
3. **注册回流**：全流程（含测试底座三类验证 + `lint_tools`）通过后，把新工具**注册回 Toolbox**，供未来复用。

> 这与文档库 §7 的"缺口→工单→人工实现"不同：默认路径是**工作流自实现**，Toolbox 提供的是"注册写回"的承接，而非等待人工补齐。仅当能力连自实现都不可能（需不存在的外部系统）时，才退回文档库 `capability_gap` 工单。

## 3. 需求清单

### R1 · 工具运行时归属（从 AIDA 迁出）
**现状**：工具硬放在 `aida/agent/tools/`，由 AIDA `DEFAULT_TOOLS` 直接持有。
**要求**：Toolbox 成为工具的**权威存放与分发处**；AIDA registry 从 Toolbox **加载**工具集，而非把实现写死在 agent 内。定义工具加载接口与迁移边界（哪些留在 AIDA 内置、哪些由 Toolbox 提供）。
**验收**：AIDA 可声明式地从 Toolbox 装载一组工具；新增工具不需改 AIDA 源码。

### R2 · 复用查询（与文档库协同）
**要求**：按能力语义查"是否已有可复用工具"——查询语义复用文档库 §6（能力词反查 + 可行性判定），Toolbox 在命中后提供**可加载的工具实体**（而非仅元数据）。
**验收**：给定能力词，能返回"是否已有工具 + 如何加载它"。

### R3 · 新工具注册 API（写侧·核心）
**要求**：工作流验收通过后，把新实现的工具注册进 Toolbox。必须支持：
- **幂等**：同一工具重复注册不产生重复项。
- **去重 / 冲突检测**：能力或工具名冲突时报错或走版本化，不静默覆盖。
- **版本化**：工具带版本号；记录 **provenance**（由哪个 skill / run / 镜像 digest 产出）。
- **目录联动**：注册即更新文档库目录该能力 `status: planned→stable`（闭合 §7）。
**验收**：注册一个新工具后，文档库能查到它、其他 skill 可复用它、provenance 可追溯。

### R4 · 工具包契约（packaging）★对齐 AIDA
**要求**：定义"一个工具"的最小可注册/可加载形态，必须同时满足：
- (a) **AIDA Tool 基类契约**（契约 §2.3）：`name`（小写下划线全局唯一）/ `description` / `parameters`（JSON Schema）/ `execute(**kwargs)` **同步**；失败**不抛异常**、返回 `"Error..."` 字符串或 `{ok:false,error}`（契约 §2.2）。
- (b) 通过 AIDA `lint_tools`（name/desc/schema 契约）与 `lint_no_naked_llm`/`lint_no_naked_send`（不得裸调 LLM / 裸外发）。
- (c) 可被**测试底座加载**（随 skill 打包，见测试底座 01 §R10）。
- (d) 可被 Toolbox **注册**（R3）。
**验收**：一个符合契约的工具包能依次通过：测试底座加载 → lint → 注册 → 被复用查询命中。

### R5 · 版本化与再验证联动
**要求**：工具有版本；skill 引用工具时记录"基于哪个工具版本"。工具升级触发**受影响 skill 的再验证**（与测试底座 §8 再验证、镜像 digest 钉死协同——工具版本进入下一次 AIDA 镜像后参与 test==prod）。
**验收**：升级一个被多个 skill 引用的工具，能列出受影响 skill 并触发再验证。

### R6 · 与 AIDA 运行时契约一致
**要求**：Toolbox 提供/注册的工具行为必须与 `AIDA-RUNTIME-CONTRACT.md` 一致（失败表达、同步 `execute`、无原生结构化输出、HITL 经 `state["hitl"]` 软中断）。外部调用类工具的端点必须**可配置**（不硬编码），以支持测试端点重定向（测试底座 01 §R10 决策）。
**验收**：注册的工具在真实 AIDA 中行为与契约一致；其外部端点可经 env/config 重定向。

### R7 · 排除妥协产物
**要求**：当前 skill-as-tool 妥协工具（`run_survey`、`zhgk_bridge` 等）**不纳入最终 Toolbox**。若确有"把某 skill 当高阶能力调用"的需求，单独设计该模式，**不混入普通工具注册路径**。
**验收**：最终 Toolbox 工具集中不含上述妥协工具；其能力若仍需要，有独立、经设计的承载方式。

### R8 · 注册前闸门
**要求**：新工具**注册前必须先通过**测试底座的三类验证（随其所属 skill）+ `lint_tools`。未过验证不得注册，避免未验证工具污染 Toolbox 被他人复用。
**验收**：构造一个未过验证的工具，注册被拒绝并给出原因。
