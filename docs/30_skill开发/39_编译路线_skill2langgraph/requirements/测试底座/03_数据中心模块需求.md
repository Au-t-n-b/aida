## 0. 定位与边界（先厘清三个易混系统）

| | 数据中心（本文） | 文档库 | Toolbox |
|---|---|---|---|
| 本质 | 运行时**文件存储 / 项目云空间** | AIDA 各模块**文档孪生** | 工具运行时 + 注册中心 |
| 存什么 | 项目文件/资产、项目、用户 | 设计文档 + 结构化能力上下文 | 工具实现/版本 |

**数据中心是 skill 运行时读写文件的地方。** 我们对它的需求只有两类：① skill 运行时**取输入 / 回传产物**；② **测试租户**所需的项目/用户。

**明确不要求数据中心提供**（属正确边界，归别处）：
- run 执行态（`state.json` / checkpoint / node 轨迹）→ Claw **容器内** SqliteSaver。
- skill 定义 + 代码 + 新工具 → AIDA **镜像**（CI 烤入）。

> v1→v2 变更：删除原 §R2/§R4/§R5 中的 `runs/<runId>/` 三级隔离、`state.json` 读 API、skill 定义层接收——那是把运行态与 skill 硬塞进云空间的越界要求。本版按真实分工收敛。

## 1. 需求清单

### R1 · 专用测试 service tenant（测试用户 + 预置已审批测试项目）★安全关键
**现状**：数据中心有人员管理（CRUD）+ 项目管理（新建走 `待审批→已审批` 状态机，审批需 `project:approve`）。
**要求**：
- 建一个**专用测试用户**（机器身份，独立于真实客户），**无真实联系人/收件人配置**（与 AIDA Agent §R9 dry-run 双保险）。
- 预置**至少一个已审批测试项目**（绕过人工审批：或给测试用户 `project:approve` 自助审批，或种子直接置 APPROVED），供测试 run 落文件。
- 测试用户/项目**不可见任何真实客户项目**（爆炸半径隔离）。
**验收**：① 测试用户可鉴权、可在测试项目下读写文件；② 其联系人为空/指向沙箱；③ 跨访问真实项目被拒。
**出处**：接口文档 §6/§7；总设计 §7。

### R2 · skill 运行时文件读写 API（确认满足 + 桥接交待）
**现状（已具备，确认即可）**：
- 取输入：`GET /files`（按 projectId/module/stage 筛选）、`GET /files/{fileId}/download`、`GET /org-assets`。
- 回传产物：`POST /files/upload`（`targetNodeId` 指定目录）。
- 定位目录：`GET /projects/{projectId}/tree`、`POST /directories`。
- 查项目详情：`GET /projects/{projectId}`（全字段 + `members[]`）。
**要求**：保证上述对**机机调用**可用（外部接口"本期免 token"利好）；其中 `GET /projects/{projectId}/tree` 标为**内部**（需登录），而上传依赖先取 tree 拿 `targetNodeId`——请确认测试身份能拿到 tree（最小权限），否则无法定位上传目录。
**桥接说明（非数据中心的活，但需对齐）**：AIDA 工具读写**本地路径**（`work_root`），数据中心是 **REST**。"下载输入到 work_root / 上传产物回云空间"的桥接由 **Manager 承接**（[02 §R2](02_Manager模块需求.md)），数据中心 files API 足以支撑，**无需新增接口**。
**验收**：测试身份能 list/download 项目文件、能 upload 产物到指定目录。

### R3 · 测试隔离按"测试项目"粒度（替代 runs/）
**要求**：并发测试 run 之间的隔离，用**独立测试项目**或**项目内独立目录**实现（数据中心已支持多项目 + `POST /directories`）。注意 `POST /files/upload` "同名按传入文件名覆盖"（接口 §5.5）——并发测试**勿共用同名输出路径**，按测试项目/目录隔离。
**验收**：并行 N 个测试 run 写产物互不覆盖。

### R4 · 业务对象 → 目录映射（供 skill 知道往哪读/写）
**现状**：节点带 `moduleCode`/`fileStage`，但"skill 的输入资料 / 最终产物"对应哪个 module/stage/目录未约定。
**要求**：提供（或与文档库 §5「C 业务对象」协同提供）"业务对象 → 目标目录"的映射，例如"项目输出"对应哪个 `moduleCode`/`fileStage` 节点，使生成的 skill 能确定读写位置。
**验收**：给定"项目输出资产"，能解析到具体可上传的目录节点。

### R5 · 测试数据清理 / TTL
**要求**：测试项目/文件有**回收策略**（TTL 或测试结束联动清理），避免测试数据堆积。注意项目"不可物理删除、仅软归档"（接口 §6）——测试项目可走归档或单独的测试数据清理通道。
**验收**：超期测试数据被清理；不波及真实项目与在跑测试。

## 2. 格式说明（不构成数据中心的交付负担）

`api_id` 稳定字符串 ID、机器可读 JSON、版本号等——是**文档库（文档孪生）**对本接口做结构化镜像时的职责，**非数据中心团队的交付**。数据中心只需保持**稳定标识符**（已有：`projectId`/`fileId`/API 路径/RBAC scope），文档库即可据此孪生供 S6 接地与编译 agent 取上下文。

## 3. 与其他模块的接缝

| 接缝 | 对端 | 说明 |
|---|---|---|
| work_root ↔ REST 文件桥接 | Manager | 下载输入到 work_root / 上传产物，由 Manager 承接（02 §R2） |
| 接口的结构化孪生（api_id/JSON）| 文档库 | 供 S6 接地、编译 agent 上下文（`文档库需求文档.md`） |
| 测试容器拉起 + 项目/目录装配 | Manager | 02 §R1/§R2 |
| 业务对象→目录映射 | 文档库 §5(C) / skill 定义 | R4 |
