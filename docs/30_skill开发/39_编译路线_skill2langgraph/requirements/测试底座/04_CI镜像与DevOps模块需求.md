# CI / 镜像 / DevOps 模块配合需求 · 测试底座

> 状态：Draft v1（2026-06-08）
> 提出方：skill2langgraph 流水线
> 面向：CI / 镜像构建 / 平台运维
> 总设计：`docs/skill2langgraph/测试底座对齐方案.md`（尤其 §4 镜像 digest=契约版本、§8 再验证流程）

## 0. 你为什么会收到这份需求

本方案"始终对齐"的最终收口是一句话：**对齐 ≡ 测试镜像 digest == 生产镜像 digest**。这把"对齐一个还在演进的 AIDA"变成"钉死一串不可变镜像快照"。要让这条成立，镜像必须不可变、test 与 prod 同源、digest 可作为版本轴贯穿产物，并在 AIDA 升级时能批量重验。这些都在 CI/镜像/运维侧。

> 落地状态提示：AIDA `04` §3 标注 **Claw 容器化 ❌待建**（当前为桌面打包）。本文是最终形态需求，请在 AIDA 镜像化时一并落地。

## 1. 需求清单

### R1 · AIDA 不可变镜像（test == prod 同源）★版本钉死基石
**要求**：把 AIDA 执行引擎（收敛后的唯一 `agent` 源，见 AIDA Agent §R1）构建为**不可变镜像**，以 **digest** 唯一标识。**测试容器与生产用户容器必须能用同一 digest**——同源构建、不存在"测试专用构建"导致行为分叉。
**验收**：① 镜像以 digest 寻址、内容寻址不可变；② 同一 digest 在测试与生产拉起的执行引擎逐字节相同；③ 构建可复现。
**出处**：总设计 §4。

### R2 · 测试租户 + 种子数据烤进测试镜像
**要求**：测试 service tenant 身份与预置已审批测试项目/输入 fixtures（数据中心 §R1）随**测试镜像**一起固化，使"测试镜像 digest"不仅锁定执行底座、**连测试身份与初始数据布局一起锁定**，完全可复现。
**说明**：测试镜像 = 生产镜像（同 R1 digest）+ 测试租户/种子层。需保证附加的测试层**不改变执行引擎行为**（仅数据/身份）。
**验收**：同一测试镜像 digest 起的容器，租户、种子项目、输入数据完全一致。

### R3 · 镜像 digest 作为契约版本贯穿产物
**要求**：流水线每个产物（生成的 Graph / 三类验证报告 / 交付包）记录"**基于哪个镜像 digest 验证通过**"。提供查询"当前生产 digest"的接口/事实源，供 Manager（§R1）与再验证流水线消费。
**验收**：任取一个已部署 Graph，能查到它"已验证通过的 digest"；能查到"当前生产 digest"。

### R4 · 再验证流水线（应对 AIDA 升级）
**要求**：当 AIDA 发布**新镜像 digest** 时，触发**无头批量重验**：对所有已部署 Graph，用新 digest 容器重跑三类验证（行为/契约/稳定性），产出"仍绿 / 转红"报告——仍绿的提升其"已验证 digest"，转红的回流编译管线或挂起。
**要求附**：重验是 CI 任务，可并行、可幂等重跑；与 Manager 并发隔离（§R5）协同。
**验收**：模拟一次 digest 升级，流水线自动对存量 Graph 出具仍绿/转红报告。
**出处**：总设计 §8。

### R5 · 回放服务部署位（确定性 LLM + 外部工具）
**要求**：部署回放服务（行为由本方案/流水线定义，CI 提供运行位），覆盖**两类非确定外部源**：
- **LLM 回放**：OpenAI 兼容端点，测试容器经 `ZHIPU_BASE_URL` 指向它（配合 AIDA Agent §R2）。
- **外部工具回放**：HTTP/MCP/数据中心 API 类工具（如 `mcp_adapter` 的 `base_url`）的回放端点，配合 AIDA Agent §R10 把这类工具重定向到回放服务。
统一要求：可按 fixture 重放、可在"录制模式"下透传真实端点并落盘录制、网络上**仅测试容器可达**。
**验收**：测试容器接入后，run 内 **LLM 调用与外部工具 HTTP 调用全部命中回放**、零真实 token、不出容器。
**出处**：总设计 §5.1/§5.2。

### R6 · fixture 版本化存储与分发
**要求**：为回放 fixtures 提供**版本化存储**，并与"测试镜像 digest / skill 版本"建立对应关系，使"哪个 digest + 哪个 skill 用哪批 fixture"可追溯、可复现。
**验收**：给定 (digest, skill, 场景)，能取到确定的一批 fixture。

## 2. 分阶段落地建议（对齐 AIDA P0→P3 路线）

| 阶段 | 本模块可先做 | 依赖 |
|---|---|---|
| P0（现在·单机无容器） | 锁 commit 代替 digest；本地起回放服务；fixture 落盘 | AIDA Agent §R1/§R2 |
| P2（容器化） | R1 不可变镜像、R2 测试镜像、R3 digest 贯穿 | AIDA 容器化、Manager §R1 |
| P3（演进自动化） | R4 再验证流水线、R6 fixture 版本化全自动 | 上述全部 |

> P0 过渡：AIDA 尚未容器化前，用 **agent 单一真相的 commit 哈希**充当版本轴（test/prod 同 commit），回放服务本地起。容器就绪后平滑切到 digest。
