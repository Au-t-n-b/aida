# 文档库需求文档（供文档库开发人员参考）

> 状态：Draft v1（2026-06-08）
> 提出方：Skill 引导编写流程 / skill 编译管线
> 面向：数据中心文档库（toolbox 清单、数据中心 API 契约、文件结构/业务对象）开发人员
> 关联设计：`docs/skill2langgraph/引导编写流程-完整设计.md`、`docs/30_skill开发/31_手写规范/AIDA-RUNTIME-CONTRACT.md`、`docs/skill2langgraph/可编译业务Skill编写规范.md`

---

## 0. 背景与目的

我们正在开发一套"把业务专家写的低确定性 skill 转成高确定性 LangGraph"的范式。其**前门**是一个引导业务人员编写合规 Raw Skill 的流程，其**主干**是编译管线。这两者在两个关键环节强依赖一个**机器可读的文档库**：

1. **写作期能力接地**（引导流程 S6）：当业务人员说"我要读 BOQ / 发通知 / 查项目详情"时，系统要能即时查到"这对应哪个业务对象、哪个工具/接口能做、能不能做"，并把不可做的暴露为能力缺口。
2. **编译期工具解析**（编译管线缺失工具策略）：把 SkillIR 里的能力需求解析为真实工具调用，或判定替代/自实现/缺口。

当前这两处在原型里是**硬编码字典**（如 `excel_reader→doc_read_xlsx`，且带 zhgk 专用注释），既不可泛化也不可维护。本文档说明：要让范式泛化到任意业务 skill，文档库需要提供什么、以什么格式提供。

> 一句话需求：**把 toolbox / 数据中心 API / 业务对象，做成一份带稳定 ID、机器可查询、可声明能力与约束的结构化清单**，让能力解析由文档库驱动，而不是由代码里的硬编码字典驱动。

---

## 1. 范围

需要文档库提供三类、可交叉引用的结构化数据：

- **A. Toolbox 工具清单**：AIDA 运行时可调用的工具（DEFAULT_TOOLS 及扩展）。
- **B. 数据中心 API 契约**：底层业务接口（项目、文件、配置等）。
- **C. 业务对象 / 文件结构**：业务人员口中的"项目资产""组织资产""配置中心"等对象及其与 A/B 的映射。

三者必须能互相引用：一个业务对象（C）通过某个 API（B）访问，并由某个工具（A）在 Graph 中调用。

---

## 2. 通用要求（三类数据共同）

1. **稳定 ID**：每个工具/接口/对象有稳定、机器可引用的 ID（英文/拼音、小写下划线）。Raw Skill、SkillIR、Graph、报告、假设清单都会引用它。
2. **机器可读**：首选 JSON / YAML（或可导出为此）。人读文档可以并存，但**必须有结构化版本**供程序查询。
3. **可查询**：支持按能力语义检索（见 §6）。引导流程要用"读 Excel""发即时消息"这类**业务能力词**反查工具，而不是让业务人员先知道工具名。
4. **声明约束与限制**：不仅说"能做什么"，还要说"做不到什么 / 有什么前置条件 / 失败长什么样"。这是判定"能力缺口"和"是否需要 HITL/config"的依据。
5. **版本化**：每类数据带版本号与更新时间；引导流程与编译管线会记录"基于哪个版本接地"，便于版本漂移时重新校验。

---

## 3. A. Toolbox 工具清单：字段要求

每个工具至少提供：

| 字段 | 必需 | 说明 |
| --- | --- | --- |
| `tool_id` | 是 | 稳定 ID，如 `doc_read_xlsx` |
| `display_name` | 是 | 中文展示名 |
| `capability_tags` | 是 | 能力语义标签数组，如 `["读取Excel","读取表格","解析BOQ"]`，供业务词反查 |
| `category` | 是 | 如 读文件 / 写文件 / 通知 / 用户交互 / 查询 / 流程 |
| `inputs` | 是 | 参数 schema（名、类型、必需性、含义） |
| `outputs` | 是 | 返回结构；**失败约定**（如返回 `"Error..."` 字符串或 `{ok:false,error}`，须与 AIDA-RUNTIME-CONTRACT 对齐） |
| `preconditions` | 是 | 前置条件，如"需 config 中的邮件服务地址" |
| `limitations` | 是 | 明确做不到的事，如"只读本地文本，不能 HTTP" |
| `side_effects` | 是 | 是否有副作用 / 是否 dry-run（如 send_mail 是否真实投递） |
| `requires_config` | 是 | 是否依赖外部 config，及 config 键 |
| `hitl_implication` | 否 | 该工具是否天然涉及人工（如 present_choices） |
| `maps_to_api` | 否 | 关联的数据中心 API（B 的 `api_id`） |
| `status` | 是 | `stable` / `experimental` / `planned`（planned 用于"尚未实现但在建"，引导流程可据此提示缺口） |

> 至少要把现有 DEFAULT_TOOLS（`read_file`、`send_mail`、`send_welink`、`present_choices`、`run_survey`、`doc_read_xlsx`、`doc_write_docx`）按此 schema 补全。这能直接替换原型里的硬编码替代表。

---

## 4. B. 数据中心 API 契约：字段要求

| 字段 | 必需 | 说明 |
| --- | --- | --- |
| `api_id` | 是 | 稳定 ID |
| `purpose` | 是 | 业务用途一句话 |
| `method` + `path` | 是 | 如 `GET /api/v1/files` |
| `request` / `response` | 是 | 参数与返回 schema |
| `business_objects` | 是 | 操作的业务对象（C 的 `object_id`），如"项目资产""项目详情" |
| `auth` / `scopes` | 是 | 鉴权与权限要求 |
| `errors` | 是 | 错误码与语义 |
| `idempotency` / `side_effects` | 是 | 写操作尤其需要 |
| `exposed_via_tool` | 否 | 哪个 toolbox 工具封装了它 |

> 现有 `数据中心对外接口设计.md`（位于上层目录）是很好的素材来源，但需要导出为带 `api_id` 的结构化版本。

---

## 5. C. 业务对象 / 文件结构：字段要求

业务人员用的是"项目资产""配置中心"这类词（见《编写规范》§7 常见来源、§19）。文档库要把这些对象结构化，作为业务语言到 API/工具的桥。

| 字段 | 必需 | 说明 |
| --- | --- | --- |
| `object_id` | 是 | 稳定 ID，如 `project_asset` |
| `aliases` | 是 | 业务人员可能用的叫法，如 `["项目资产","项目文件","项目资料"]` |
| `description` | 是 | 是什么 |
| `access_via_api` | 是 | 读/写它用哪些 API（B 的 `api_id`） |
| `typical_artifacts` | 否 | 常见承载物（BOQ.xlsx、勘测预置集.docx 等）及格式 |
| `relations` | 否 | 与其他对象的关系（项目 → 项目资产 → 文件） |

至少覆盖《编写规范》§7 列出的来源：项目资产、组织资产、项目详情、配置中心、用户上传、人工确认、上一步输出。

---

## 6. 查询能力要求（引导流程 / 编译管线如何用）

文档库需对外提供（API 或本地可加载索引）至少以下查询：

1. **能力词 → 工具**：输入"读取 Excel"，返回候选工具（按 `capability_tags` 匹配，带相似度）。供引导流程 S6 反查。
2. **业务对象别名 → 对象 → 访问方式**：输入"项目资产"，返回 `object_id` 及读写它的 API/工具。
3. **工具/接口可行性判定**：输入一个能力需求，返回 `available` / `needs_config` / `experimental` / `planned` / `not_found`。`planned`/`not_found` 即触发能力缺口。
4. **失败与约束查询**：返回某工具的 `limitations` 与失败约定，供生成正确的错误/重试/warning 语义。

> 这些查询直接替换原型里 `_SUBSTITUTIONS` / `_SELF_IMPLEMENTATIONS` 两张硬编码字典。替代关系应能由"能力词 → 多候选工具 + 各自 limitations"自然推导，而非写死。

---

## 7. 缺口反馈机制（双向）

引导流程与编译管线会**产出文档库的需求**，需要一条回流通道：

- 当能力需求查不到可用工具（`planned`/`not_found`），引导流程把它记成假设清单里的 `capability_gap`，并生成一条**文档库/工具需求工单**，至少含：能力描述、来源 skill、业务影响、期望语义、是否阻断。
- 建议文档库提供一个轻量收口（哪怕是结构化文件目录 `requests/` + 模板），让这些缺口能被工具开发团队消费。
- 闭环：工具/接口实现后，更新清单 `status` 由 `planned→stable`，引导流程/编译管线据版本变更重新校验受影响的 skill。

> 与 Toolbox 模块的分工（见 `Toolbox模块需求.md`）：能力查不到时，**默认路径不是等人工补齐，而是编译管线自行实现新工具**，跑完经 Toolbox 注册回流——注册时由 Toolbox 联动把本目录 `status` 置 `stable`。本节的"工单"仅用于**连自实现都不可能**（需不存在的外部系统）的真·缺口。文档库在此只承担"目录/索引（读侧）"，工具实体的注册写回属 Toolbox。

---

## 8. 与 AIDA 运行时契约的一致性约束

文档库声明的工具行为**必须与** `AIDA-RUNTIME-CONTRACT.md` 一致，重点：

- 工具失败表达：`"Error..."` 字符串或 `{ok:false,error}`。
- HITL 表达：经 `state["hitl"]` 软中断后路由 END；涉及人工的工具要在 `hitl_implication` 标注。
- 结构化输出走 adapter 提示 + JSON 解析 + 重试，而非工具原生 schema——故工具 `outputs` 应描述实际返回，不假设 provider 原生结构化。
- 默认工具白名单（DEFAULT_TOOLS）应在清单中显式标注，便于"是否需要替代/自实现"的判定。

---

## 9. 优先级（建议交付顺序）

1. **P0**：A（Toolbox 清单）按 §3 schema 把现有 DEFAULT_TOOLS 七个工具补全 + §6 查询 1/3（能力词反查、可行性判定）。这两项一就绪，引导流程 S6 即可摆脱硬编码。
2. **P1**：C（业务对象）覆盖《编写规范》§7 来源 + §6 查询 2。
3. **P1**：B（API 契约）从 `数据中心对外接口设计.md` 结构化导出。
4. **P2**：§7 缺口反馈通道 + §2 版本化。

---

## 10. 给文档库开发的最小验收清单

- [ ] 每个工具/接口/对象有稳定 ID。
- [ ] 存在机器可读（JSON/YAML）版本，非仅人读文档。
- [ ] 工具带 `capability_tags`，支持业务能力词反查。
- [ ] 工具/接口声明 `limitations`、失败约定、`requires_config`。
- [ ] 业务对象带 `aliases` 且映射到 API/工具。
- [ ] 提供 §6 的四类查询（至少 P0 的两类）。
- [ ] 工具行为与 AIDA-RUNTIME-CONTRACT 一致。
- [ ] 带版本号与更新时间。
- [ ] 有 `planned` 状态以表达"在建/未实现"，支撑能力缺口检测。
