# 可编译业务 Skill 编写规范

> 适用对象：业务人员、流程负责人、产品经理、交付专家，以及需要把业务流程交给 skill2langgraph 固化为 LangGraph 流水线的人员。
>
> 本规范定义当前工作流所需 Raw Skill 的规格。Raw Skill 不是代码实现文档，而是可编译的业务流程契约。

## 1. 核心定位

Raw Skill 的目标是让不懂代码的业务人员只描述：

- 要完成什么业务目标
- 需要哪些输入
- 产出什么结果
- 中间有哪些步骤
- 哪些规则必须遵守
- 缺信息时怎么处理
- 哪些地方需要模型判断
- 哪些地方需要人工观察者确认或补料
- 什么情况算成功或失败

skill2langgraph 会把 Raw Skill 转成：

```text
Raw Skill
-> SkillIR
-> Graph / GraphSpec
-> 测试
-> Test Harness 执行
-> Graph 文档和检测报告
```

Raw Skill 作者不需要写 Python、LangGraph、HTTP 接口或工具实现。

## 2. 编写原则

### 2.1 写业务契约，不写代码

应该写：

```text
如果 BOQ 中液冷设备数量大于 0，则场景为液冷工勘。
```

不要写：

```text
调用 filter_scene() 函数判断。
```

### 2.2 写输入输出，不写实现细节

应该写：

```text
输入：BOQ 文件，xlsx 格式，缺失时进入补料。
输出：风险识别表，xlsx 格式，包含等级、描述、建议动作。
```

不要写：

```text
用 pandas 读取 Excel，然后写入 openpyxl。
```

### 2.3 写工具能力，不强行指定工具名

应该写：

```text
需要读取 Excel 文件。
需要上传输出文件到项目资产。
需要发送 WeLink 通知。
```

如果业务人员知道工具名，可以写为“建议工具”，但不应把工具名作为业务正确性的唯一依据。

### 2.4 写可验证标准

每个输出和关键步骤都要能被测试。

例如：

```text
报告必须包含项目概况、场景判断、风险列表和建议动作。
缺 BOQ 时必须触发补料，不得继续生成正式报告。
```

### 2.5 明确不确定性

如果某个判断不能完全规则化，要写清楚：

```text
该步骤需要模型根据资料判断，并给出判断依据。
```

或者：

```text
该步骤需要观察者确认。
```

## 3. 推荐文档结构

Raw Skill 应按以下结构编写：

```text
# Skill: <业务名称>

## 1. 基本信息
## 2. 业务目标
## 3. 适用范围与非目标
## 4. 输入
## 5. 输出
## 6. 流程步骤
## 7. 业务规则
## 8. 人工观察者 / HITL
## 9. 工具能力需求
## 10. LLM 使用点
## 11. 状态字段
## 12. 错误、重试与警告
## 13. 成功标准
## 14. 测试场景
## 15. 示例输入输出
```

## 4. 基本信息

必须包含：

```yaml
name: zhgk
title: 智慧工勘
version: 1.0
owner: 交付业务组
domain: 数据中心交付
description: 根据项目资料生成工勘评估报告，并识别风险与待补充项。
```

字段说明：

| 字段 | 要求 |
| --- | --- |
| `name` | 稳定英文或拼音 ID，用于 graph、run、report、config |
| `title` | 中文展示名称 |
| `version` | 业务流程版本 |
| `owner` | 业务负责人或团队 |
| `domain` | 业务领域 |
| `description` | 一句话说明业务目的 |

## 5. 业务目标

用一段话回答：

```text
这个 Skill 要把什么输入，经过什么业务流程，变成什么输出？
```

示例：

```text
根据 BOQ、勘测信息预置集、项目基础资料，完成智慧工勘场景筛选、勘测信息汇总、风险识别和工勘报告生成。
```

## 6. 适用范围与非目标

必须写清楚能做什么和不做什么。

示例：

```text
适用范围：
- 数据中心智慧工勘场景
- 项目资料已具备基础 BOQ
- 输出工勘报告、风险表、分发建议

非目标：
- 不负责正式审批流
- 不直接修改项目主数据
- 不替代人工现场勘测
```

## 7. 输入定义

输入应使用表格。

| 输入 | 是否必需 | 来源 | 格式 | 缺失处理 |
| --- | --- | --- | --- | --- |
| BOQ 文件 | 必需 | 项目资产 | `.xlsx` | 进入补料 HITL |
| 勘测信息预置集 | 必需 | 项目资产 | `.docx` | 进入补料 HITL |
| 项目基本信息 | 必需 | 数据中心项目详情 | JSON | 自动查询；查不到则失败 |
| 审批人列表 | 可选 | 配置中心 / config | JSON | 缺失则警告 |

规范：

- 每个输入必须有稳定名称。
- 必须说明是否必需。
- 必须说明来源。
- 必须说明格式。
- 必须说明缺失处理。
- 来源可以写业务对象，不必写具体 API。

常见来源：

```text
项目资产
组织资产
项目详情
配置中心
用户上传
人工确认
上一步输出
```

## 8. 输出定义

输出也应使用表格。

| 输出 | 类型 | 归属 | 用途 | 成功标准 |
| --- | --- | --- | --- | --- |
| 工勘报告 | `.docx` | 项目输出资产 | 提交评审 | 文件存在，且包含风险章节 |
| 风险识别表 | `.xlsx` | 项目输出资产 | 后续跟踪 | 包含等级、描述、建议动作 |
| 执行摘要 | Markdown / JSON | run report | 给观察者查看 | 说明完成情况和警告 |

规范：

- 输出必须可验证。
- 输出必须说明用途。
- 输出必须说明归属。
- 输出失败但流程可继续时，要明确是 warning 还是失败。

## 9. 流程步骤

流程步骤是 Raw Skill 最重要的部分。

推荐表格：

| 步骤 ID | 步骤名称 | 目标 | 输入 | 输出 | 失败/缺失处理 |
| --- | --- | --- | --- | --- | --- |
| preflight | 环境预检 | 检查项目资料是否齐备 | 项目资料 | 齐备检查结果 | 缺文件则补料 |
| scene_filter | 场景筛选 | 判断适用工勘场景 | BOQ、项目资料 | 场景类型 | 无法判断则人工确认 |
| survey_build | 勘测汇总 | 生成勘测信息汇总 | 勘测预置集 | 勘测摘要 | 缺字段则生成警告 |
| report_gen | 报告生成 | 生成工勘报告 | 前序结果 | docx 报告 | 失败则重试 |
| report_distribute | 报告分发 | 生成分发建议 | 报告、人员配置 | 分发摘要 | 缺联系人则警告 |

规范：

- 每个步骤必须有稳定 `步骤 ID`。
- 步骤 ID 使用小写英文、数字和下划线。
- 每步只做一个清晰业务动作。
- 每步必须写输入、输出和异常处理。
- 默认按表格顺序执行。
- 如果有分支，必须显式说明条件。

## 10. 业务规则

业务规则应列表化，不要藏在长段落里。

示例：

```text
- 如果 BOQ 中液冷设备数量 > 0，则场景为“液冷工勘”。
- 如果配电负载率 > 80%，风险等级为“高”。
- 如果缺少 CAD 图纸，不阻断报告生成，但必须写入 warning。
- 如果缺少 BOQ，不得进入报告生成步骤。
```

规范：

- 能量化的规则必须量化。
- 模糊判断必须标注“需要 LLM 判断”或“需要人工确认”。
- 每条规则应能被测试或被人工审阅。

## 11. 人工观察者 / HITL

当前工作流中的人类是 observer，不参与编辑。

可以允许：

```text
- 查看当前步骤
- 查看缺失输入
- 上传补充文件
- 选择继续 / 重试 / 终止
- 查看最终报告和警告
```

不允许：

```text
- 直接编辑中间状态
- 手动修改 Graph
- 手动修改测试结果
- 覆盖最终检测结论
```

HITL 类型建议：

| 类型 | 触发条件 | 行为 |
| --- | --- | --- |
| 文件补料 | 必需输入缺失 | 提示上传文件 |
| 业务确认 | 多个可选路径 | 让观察者选择 |
| 失败恢复 | 步骤失败 | 允许重试或终止 |
| 警告确认 | 非阻断问题 | 展示但继续 |

## 12. 工具能力需求

Raw Skill 应写业务能力，不必写工具实现。

示例：

```text
需要的工具能力：
- 读取 Excel 文件
- 读取 Word 文件
- 写入 Word 报告
- 上传输出文件到项目资产
- 查询项目详情
- 查询项目文件列表
- 发送 WeLink 通知
```

如果已知工具名，可以写：

```text
建议工具：
- doc_read_xlsx：读取 BOQ
- doc_write_docx：写报告
```

但最终工具选择由 skill2langgraph 根据 toolbox 清单、数据中心 API 和缺失工具策略决定。

## 13. LLM 使用点

必须明确哪些步骤需要模型。

| 步骤 | 是否需要 LLM | 用途 | 输入 | 输出要求 |
| --- | --- | --- | --- | --- |
| scene_filter | 是 | 判断工勘场景 | BOQ 摘要、项目资料 | 场景类型 + 理由 |
| survey_build | 是 | 汇总勘测信息 | 文档内容 | 结构化摘要 |
| report_gen | 是 | 生成报告正文 | 所有前序结果 | Markdown 或报告内容 |

规范：

- 不要让所有步骤默认用模型。
- 能用规则完成的，不交给模型。
- LLM 输出必须有结构要求。
- 高风险判断必须要求给出依据。

## 14. 状态字段

业务人员可以用简单结构说明流程中需要记住什么。

示例：

```yaml
state:
  project:
    project_id: string
    project_name: string
  inputs:
    boq_file: file
    survey_preset: file
  intermediate:
    scene_type: string
    risks: list
    survey_summary: object
  outputs:
    report_docx: file
    risk_table: file
  hitl:
    step: string
    reason: string
```

规范：

- 至少要说明项目、输入、中间结果、输出、HITL。
- 不要求业务人员写类型系统，但要让生成器知道哪些信息需要跨步骤保存。

## 15. 错误、重试与警告

必须说明常见异常如何处理。

示例：

| 情况 | 处理 |
| --- | --- |
| 必需文件缺失 | 触发补料 HITL |
| 可选联系人缺失 | 生成 warning，流程继续 |
| LLM 输出无法解析 | 重试，仍失败则进入错误状态 |
| 工具 config 缺失 | 返回错误，并提示 agent 引导用户补 config |
| 外部通知失败 | 记录 warning 或失败，按业务重要性决定 |

规范：

- 必需输入缺失通常进入 HITL。
- 可选信息缺失通常生成 warning。
- 影响最终输出正确性的错误应失败。
- 允许重试的步骤必须说明重试范围。

## 16. 成功标准

成功标准是测试生成的重要依据。

示例：

```text
成功标准：
- 所有必需步骤完成。
- 工勘报告文件存在。
- 风险识别表文件存在。
- 如果输入完整，不应触发补料 HITL。
- 如果缺 BOQ，必须触发补料 HITL。
- 最终报告必须包含：项目概况、场景判断、风险列表、建议动作。
```

规范：

- 每条成功标准应可验证。
- 不要只写“结果正确”。
- 必须覆盖输出、HITL、warning 和失败。

## 17. 测试场景

至少提供四类测试场景：

| 场景 | 输入 | 期望 |
| --- | --- | --- |
| 正常输入完整 | BOQ + 勘测预置集 | 全流程完成 |
| 缺必需输入 | 只有勘测预置集 | 触发补料 |
| 业务风险场景 | BOQ 中负载过高 | 报告包含高风险 |
| 非阻断 warning | 无审批人配置 | 生成 warning 但不中断 |

规范：

- 至少一个 happy path。
- 至少一个缺失输入场景。
- 至少一个业务规则分支。
- 至少一个 warning 场景。
- 如有 HITL，必须提供 HITL 场景。

## 18. 示例输入输出

建议提供简化示例。

示例输入：

```text
项目：K1903 智算一期
BOQ：包含液冷机柜、配电柜、网络设备
勘测预置集：包含机房面积、层高、供电信息
```

示例输出：

```text
场景类型：液冷工勘
风险：
- 高：配电负载率超过 80%
- 中：桥架路径未确认
输出文件：
- 工勘报告.docx
- 风险识别结果表.xlsx
```

## 19. 数据中心对象写法

如果 Skill 涉及项目、文件、目录、用户，应使用数据中心业务对象描述。

参考：

```text
docs/数据中心API调用规范.md
```

业务人员可以写：

```text
从项目资产中读取 BOQ 文件。
把生成的报告上传到项目输出目录。
查询项目详情获取项目名称、客户名称、成员角色。
```

不需要写：

```text
GET /api/v1/files
POST /api/v1/files/upload
```

接口映射由 skill2langgraph 和 toolbox integration 处理。

## 20. 禁止写法

Raw Skill 中应避免：

```text
- 直接要求调用某个 Python 函数。
- 只写“自动处理”，不说明步骤。
- 只写“生成报告”，不说明报告内容。
- 只写“失败时重试”，不说明重试范围。
- 把工具缺失当作不存在。
- 把人工观察者写成可以手动编辑 Graph 或测试。
- 把模型判断写成绝对事实，但不给依据要求。
- 把输入输出藏在长段落里。
```

## 21. 最小合格 Raw Skill 检查清单

提交 Raw Skill 前检查：

- 有稳定 `name`。
- 有明确业务目标。
- 有适用范围和非目标。
- 输入表完整。
- 输出表完整。
- 流程步骤表完整。
- 至少列出关键业务规则。
- 写明 HITL 行为。
- 写明工具能力需求。
- 写明 LLM 使用点。
- 写明状态字段。
- 写明错误、重试和 warning。
- 写明成功标准。
- 至少提供四类测试场景。
- 不包含代码实现细节。
