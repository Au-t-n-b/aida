# AI 智能化系统架构梳理

## 1. 系统定位

整套系统不是传统的信息化排期系统，而是基于 AI，尤其是大模型和 Agent 构建的新一代智能化交付系统。

它的目标不是简单把业务流程线上化，而是让系统具备交付领域理解能力、业务语义建模能力、角色化推理能力和行动编排能力，从而支撑 PD、TD 等角色更好完成算力交付履约。

可以将系统理解为：

以企业交付知识为基础，以 DORA 本体为语义骨架，以交付 Claw / Agent 为执行入口，以算力交付履约为业务目标的 AI 原生交付编排系统。

## 2. 总体架构

当前设计可以分为四层：

1. Wiki 大脑：沉淀企业交付领域文档和上下文知识。
2. DORA 本体建模：把交付业务抽象为 Decision、Object、Relation、Action。
3. 交付 Claw / Agent：按角色和任务构建交付场景 Agent。
4. 交付编排应用：面向 PD、TD 等角色支撑排期、调整、风险识别和协同。

整体关系是：

企业文档 -> Wiki 大脑 -> DORA 本体 -> 交付 Claw / Agent -> 交付编排应用 -> 交付履约动作

其中，Wiki 大脑负责让系统“懂业务资料”，DORA 本体负责让系统“懂业务结构”，交付 Claw 负责让系统“能按角色完成任务”，交付编排应用负责让能力进入实际交付工作流。

## 3. Wiki 大脑

### 3.1 定位

Wiki 大脑是企业级交付知识底座。

交付领域存在大量企业文档，包括交付规范、项目材料、计划模板、技术方案、验收要求、风险经验、角色职责和历史案例。这些文档不能只作为附件存放，而需要被结构化整理为可检索、可引用、可被 Agent 消费的知识系统。

基于这些文档构建 llm-wiki，可以形成一个企业级 Wiki 大脑。

### 3.2 作用

Wiki 大脑主要服务三类消费方：

- DORA 本体建模：访谈过程中需要补充业务上下文时，检索 Wiki 大脑，帮助抽取 Decision、Object、Relation、Action。
- 交付 Claw / Agent：Agent 执行具体任务时，检索 Wiki 大脑，获取业务规则、流程、风险和解释依据。
- 业务用户：在需要解释计划、风险或动作原因时，系统可以引用 Wiki 中的业务依据。

### 3.3 价值

Wiki 大脑的关键价值是把企业文档从“静态资料”变成“可被智能系统调用的业务上下文”。

它解决的问题包括：

- 业务知识散落在文档中，难以被统一使用
- Agent 缺少企业上下文，只能进行通用推理
- 本体访谈容易停留在表层，难以挖出真实业务对象和关系
- 输出解释缺少可追溯依据

## 4. DORA 本体建模

### 4.1 定位

DORA 本体建模是系统的业务语义层。

这套本体建模深度参考 Palantir 的本体论思想，但在当前场景中需要落到交付业务本身：不仅描述有哪些对象，还要描述业务决策、对象关系和可执行动作。

DORA 指：

- Decision：业务决策，系统和角色需要判断什么
- Object：业务对象，交付场景中被管理和被操作的对象
- Relation：业务关系，对象之间如何连接、依赖和影响
- Action：业务动作，角色或 Agent 可以触发什么操作

### 4.2 两步构建方式

当前本体建模包括两个前后衔接的阶段。

第一步是业务上下文结构化构建。

也就是先构建 Wiki 大脑，把企业交付文档转化为可检索、可引用、可被 Agent 使用的上下文。没有这一步，DORA 访谈容易只依赖专家当场表达，系统也难以体现“懂业务”。

第二步是基线本体构建。

采用 DORA 访谈模式，由 Agent 与业务专家沟通。在对话过程中，如果涉及业务知识，Agent 会检索 Wiki 大脑，把相关业务上下文补充进来，再继续追问和抽取本体信息。

从业务专家体感上，这种方式应该让专家感到：

- 系统不是泛泛提问，而是理解当前交付业务背景
- 访谈能围绕真实业务对象、决策和动作展开
- Agent 能主动补充企业文档中的上下文
- 本体建模不只是整理词汇，而是捕获业务运行逻辑

### 4.3 本体丰富

有了基础本体之后，还需要继续面向更贴近一线操作的业务专家做深入 DORA。

这个阶段的重点不是重复确认基础对象，而是捕获实际工作中的处理逻辑，例如：

- PD 遇到客户机房延期时如何判断影响
- TD 遇到安装或调试异常时如何判断后续动作
- 设备到场、入场窗口、人力资源之间如何权衡
- 哪些风险需要升级，哪些可以局部协调
- 计划调整后如何解释给客户和内部团队

这些内容会让本体从“基础语义模型”进一步变成“可支持业务推理和行动建议的业务操作模型”。

## 5. 交付 Claw / Agent

### 5.1 定位

交付 Claw 是面向交付场景的 Agent 系统。

它参考 OpenClaw、LangGraph 等 Agent harness 框架思想，构建自有的交付 Agent 体系。这里的重点不是单个聊天助手，而是按交付角色、交付任务和业务动作组织 Agent。

### 5.2 角色化 Agent

后续可以按照交付角色划分 Agent，例如：

- PD Agent：面向项目管理、计划调整、风险协同和客户沟通
- TD Agent：面向技术交付、安装调试、点亮测试和技术风险判断
- 计划 Agent：面向排期生成、计划调整、沙箱推演和影响分析
- 风险 Agent：面向约束冲突识别、风险解释和升级建议
- 知识 Agent：面向 Wiki 检索、依据引用和业务问答

不同 Agent 应挂载不同 Skill，并根据角色消费 Wiki 大脑和本体。

### 5.3 Agent 对 Wiki 和本体的消费

交付 Claw / Agent 不应只依赖大模型自身知识，而要显式消费企业 Wiki 和业务本体。

典型方式包括：

- 从 Wiki 大脑检索企业文档、规则、案例和解释依据
- 从 DORA 本体读取业务对象、关系、动作和决策点
- 根据角色 Skill 选择合适的工具和工作流
- 在排期调整、风险识别、客户解释等任务中生成可追溯输出

因此，Agent 的能力来自三部分组合：

大模型推理能力 + 企业 Wiki 上下文 + DORA 本体结构

## 6. 与算力交付编排系统的关系

算力交付编排系统是 AI 智能化能力进入业务现场的应用层。

它面向 PD、TD 等角色，解决的是：

- 货到了以后，现场能不能干
- 什么时候能干
- 谁去干
- 先干什么，后干什么
- 出了问题谁协调
- 计划变化后如何调整原先排期
- 调整方案在沙箱中执行后有哪些风险
- 当前计划是否还能支撑合同按时交付

底层的 Wiki 大脑、DORA 本体和交付 Claw 分别提供：

- Wiki 大脑：提供交付领域知识和依据
- DORA 本体：提供业务对象、关系、决策和动作结构
- 交付 Claw：提供面向角色的 Agent 执行能力

上层的交付编排应用负责把这些能力组织成用户可用的工作流和界面。

## 7. 闭环机制

这套系统应该形成持续迭代闭环。

### 7.1 知识闭环

企业文档进入 Wiki 大脑，Wiki 大脑被 DORA 和 Agent 消费；新的交付经验、风险解释和专家修正，也应反向沉淀回知识体系。

### 7.2 本体闭环

DORA 访谈生成基础本体；实际交付中暴露出的新对象、新关系、新动作和新决策逻辑，应继续丰富本体。

### 7.3 Agent 闭环

Agent 在执行交付任务时暴露出的能力缺口，应沉淀为新的 Skill、工具或工作流。

### 7.4 业务闭环

排期、调整、沙箱推演、风险识别和协同动作产生的结果，应持续反馈给 PD、TD 和业务专家，帮助系统从“能回答”逐步走向“能协同完成交付”。

## 8. 一句话总结

整套系统的智能化主线是：

用 Wiki 大脑沉淀企业交付知识，用 DORA 本体构建业务语义骨架，用交付 Claw / Agent 承接角色化任务执行，最终支撑 PD、TD 等角色按合同要求完成算力交付履约。
