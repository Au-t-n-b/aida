# 校验规则集（S8 与 Session 0 同源 · 草案 v0.3）

> 状态：Draft v0.3（2026-06-09）· **未冻结**
> v0.2→v0.3（walking skeleton 实跑回填）：新增 R-A-06（自定义跨步态须 binding=runtime_file，否则运行时丢）、R-K-09（LLM 不作统计真源）、R-C-18（打包须含 SKILL.md+module.json）；R-K-01 把 `runtime_file` 落盘视为合法产出。
> v0.1→v0.2（device_install 压测回填）：R-C-10 改为"语义覆盖"修误报(G5)；R-K-01 把 `hitl.writes` 计入产出修误报(G2)；新增 R-C-15/16/17（self_impl 契约/集合产物/requires_llm）与 R-A-05（form_submit writes）。
> 定位：把"完整性 + 一致性 + 运行时契约一致"表达成**对 SkillIR 可机器执行的断言**。**一份定义，两处调用**——前门 **S8 HARD-GATE**（写作期，失败内联回 S7 定点补问）与下游 **Session 0**（复核，失败打回），避免双实现漂移（开放问题 §12）。
> 校验对象：`SkillIR-schema.md` 定义的 SkillIR（部分完整性规则也对 Raw Skill §22 生效）。
> 严重度：`error`=阻断（不通过不放行）；`warn`=放行但记录。

## 0. 闸门纪律（照搬 brainstorming HARD-GATE）

- S8 通过且 S9 作者批准前，**不产出最终 Raw Skill、不移交编译**。
- 反模式应答："这太简单不用写这么细"——越简单越易藏未审假设；**最小产物也要过闸门**，只是篇幅可短。
- **绝不带病放行**：任一 `error` 未清不通过。

## 1. 完整性校验（对应《编写规范》§21）

| ID | 严重度 | 断言（对 SkillIR） | 失败提示模板 |
|---|---|---|---|
| R-C-01 | error | `meta` 含 name/title/version/owner/domain/description，且 `name` 匹配 `^[a-z][a-z0-9_]*$` | "缺基本信息：{字段}" |
| R-C-02 | error | `scope.in_scope` 与 `scope.non_goals` 均非空 | "未写适用范围/非目标" |
| R-C-03 | error | `inputs[]` 每项含 id/required/source/format/on_missing | "输入 {id} 缺字段 {x}" |

| R-C-05 | error | `steps[]` 每项含 id/goal/depends_on/reads/writes/kind/on_fail；`on_fail=retry` 时 `retry_scope` 非空 | "步骤 {id} 缺字段/缺重试范围" |

| R-C-10 | error | `state` **语义覆盖**输入/中间/输出/hitl 四类各有归属字段（**组名不限**，允许 `session`/`tasks_state` 等自定义分组，G5） | "状态缺 {语义类} 的字段归属" |

## 2. 一致性 / 可满足性校验（对应前门设计 §7.2 · 旧管线完全缺失）

这是比"缺失"更隐蔽、必须机器检测的一类。

| ID | 严重度 | 断言（formal） | 失败提示模板 |
|---|---|---|---|
| **R-K-01 步骤输入可达** | error | ∀ step s，∀ r ∈ s.reads：r ∈ (declared `inputs.id`) ∪ (`state` 声明字段，含 `binding=runtime_file` 的落盘态，v0.3) ∪ (⋃ 上游 `step.writes`) ∪ (⋃ 上游 `hitl.writes`，即编辑器提交 payload，G2)，上游 = s.depends_on 闭包 | "步骤 {s} 的输入 {r} 没有任何上游产出它——漏了一步，还是它其实是输入/编辑器提交？" |
| **R-K-02 成功标准可映射** | error | ∀ sc ∈ success_criteria：sc.maps_to 解析到某 `output.id` / `state` 字段 / `step.id` 状态 | "成功标准 {sc} 落不到可验证产物，不能只写'结果正确'" |
| **R-K-03 状态字段已声明** | error | ∀ step 的 reads/writes 中引用的 `state` 字段 ∈ `state` 声明 | "步骤 {s} 引用了未声明的状态字段 {f}" |
| **R-K-04 HITL 触发有据** | error | ∀ hitl h：h.trigger 引用的 input/step 真实存在 | "HITL {h} 的触发条件指向不存在的 {ref}" |
| **R-K-05 失败处理不自相矛盾** | error | 不存在：某 `input.on_missing ∈ {hitl_supply,fail}`（缺它不得继续）∧ 某 success_criteria 在该输入缺失路径下仍要求其下游产物 | "缺 {input} 进补料/不得继续，却有成功标准要求其产物——冲突" |
| **R-K-06 依赖无环** | error | `steps` 的 `depends_on` 构成 DAG（无环） | "步骤依赖成环：{cycle}" |
| **R-K-07 LLM 点有结构要求** | error | ∀ llm_point：`output_schema` 非空；∀ business_rule(kind=llm)/高风险判断：对应 llm_point `requires_rationale=true` | "LLM 点 {p} 未声明输出结构 / 高风险判断未要求依据" |
| **R-K-08 可追溯** | warn | 凡被收紧的点（阈值 rule、分支 step、capability_substituted/gap）带 `assumption_ref` | "{元素} 是收紧决定但无假设回指" |
| **R-K-09 统计真源非 LLM** | error | 凡可量化统计（计数/数量/比率）用作 success_criteria 或写入产物：真源须 **rule 步骤派生**，不得以 `llm_points` 输出字段为准（LLM 仅可复述）。v0.3·实跑教训：LLM 写"40"实为 29 | "统计量 {x} 以 LLM 文本为真源，须改取规则派生值" |

## 3. 运行时契约一致（对齐 AIDA-RUNTIME-CONTRACT）

| ID | 严重度 | 断言 | 失败提示 |
|---|---|---|---|
| R-A-01 工具取值域 | error | ∀ tool_need(resolution=reuse)：`tool_id` ∈ toolbox 可用工具集；否则 resolution 必为 `self_impl`/`gap` | "工具 {id} 不在工具集且未声明自实现/缺口" |
| R-A-02 结构化在适配器层 | error | llm_points 的 `output_schema` 是"适配器解析目标"，IR 不假设 provider 原生结构化输出 | "LLM 点假设了原生 output_schema" |
| R-A-03 HITL 软中断 | error | hitl 条目以 `trigger`+`resume_mode` 表达（写 state→END→resume），不出现 `interrupt()` 语义 | "HITL {h} 用了非软中断语义" |
| R-A-04 state diff/reducer | warn | step.writes 语义为"返回 diff"；`steps`/`logs` 累加由运行时保证，IR 不写整份覆盖 | "步骤 {s} 疑似整份 state 覆盖" |
| R-A-05 form_submit 落盘 | error | ∀ hitl(type=form_submit)：有 `writes`（提交 payload 经 `apply_resume_payload` 并入 state，G2） | "form_submit {h} 未声明 writes，下游读不到提交结果" |
| **R-A-06 跨步态绑定** | error | ∀ 跨步自定义 state 字段：`binding=runtime_file`（落盘）；只有 SkillState 内置 channel（run_id/skill_id/project/steps/logs/current_step/overall_progress/files/hitl/hitl_resume/metrics/error）可标 `binding=channel`。否则运行时 LangGraph 丢弃 diff（v0.3·实跑发现） | "状态 {f} 标了 channel 但非内置键，运行时会丢——应 runtime_file" |

## 4. 同源调用（两处，一份）

| 调用方 | 时机 | 通过后 | 失败动作 |
|---|---|---|---|
| **S8 HARD-GATE** | 前门写作期 | → S9 作者终审 | 内联**回 S7 定点补问**（用失败提示模板逐条问作者），绝不放行 |
| **Session 0** | 编译管线入口 | → 编译（会话 A–D） | **复核打回**（因 S8 已跑过同一套，此处多为确认；命中即退回） |

> 实现要求：本规则集是**单一可执行模块**（一份规则定义），S8 与 Session 0 各自调用同一实现，杜绝双实现漂移（§12）。建议产出 `lint_skill_contract`（对 Raw Skill/SkillIR）与 AIDA 既有 `lint_runtime_contract` 协同。

## 5. 输出格式（机器可消费）

校验产出结构化报告，供 S8 内联展示 / Session 0 判定 / CI 消费：

```jsonc
{
  "passed": false,
  "errors":  [ { "rule": "R-K-01", "target": "steps.survey_build.reads.survey_summary", "message": "…没有上游产出它…" } ],
  "warnings":[ { "rule": "R-K-08", "target": "business_rules.rule_high_load", "message": "收紧决定无假设回指" } ],
  "summary": { "error_count": 1, "warn_count": 1 }
}
```

`passed = (error_count == 0)`。
