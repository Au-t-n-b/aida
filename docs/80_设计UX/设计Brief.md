# AIDA 前端设计 Brief（交付 Claude Design）

> 本文档是交给 Claude Design 进行界面设计前的**意图对齐文档**，沉淀已达成共识的设计原则、信息架构、主页方案、视觉规范与待定项。
> 读者：Claude Design 及后续原型/视觉设计者。
> 配套阅读：[页面设计偏好](页面设计偏好.md)（领导视角原始偏好）、[DESIGN.md](DESIGN.md)（设计 token 真相）。前端实现在本仓 [`frontend/`](../../frontend/)；业务背景与 Logo 在 `aida-delivery` 知识库。

---

## 1. 系统定位与气质

- 系统品牌：**AIDA**（Agentic Integrated Delivery Arena，智能体集成交付架构）。
- 业务定位：面向算力交付的**履约编排系统**，以合同交付目标为终点，以"人、货、站"为核心约束，以 PoD/机房为组织粒度。
- 底层是 AI 原生四层架构：Wiki 大脑 → DORA 本体 → 交付 Claw/Agent → 交付编排应用。**前端是这套智能化能力的唯一"出口"**。
- 整体气质：企业级、冷静、精确、有结构感、可信。**不要**炫酷 AI 图标、机器人、赛博/蓝紫渐变、电路板。可信度来自结构而非特效。
- 关键前提：系统会**面向顶层领导汇报**，主页起就要有表达力和冲击力，让人一眼看到"这套系统在替我盯整个交付盘子，而且比人看得更准"。

---

## 2. 核心设计原则（已锁定）

1. **态势优先，而非工具优先**：先让用户看到交付态势与 AI 研判，对话/工具是手段不是入口。
2. **信息逐级披露**：每一层只露宏观信息 + 关键信息 + 异常；非重点默认收起，点击/展开/下钻才深入，越钻越细。
3. **AI 可见即可信**：系统的"聪明"必须在 UI 上有固定表达载体——每条研判可溯源（依据/推理），排期调整必须展示对照。详见第 7 节。
4. **一套数据、多个镜头**：同一份人货站数据，通过顶部视角切换器在全局视角/PD/TD 之间切换，各自突出不同关注点（领导不直接操作系统，通过全局视角查看）。详见第 5 节。
5. **企业级克制视觉**：红黑灰为主色，蓝色克制，结构感优先于特效。详见第 9 节。

---

## 3. 信息架构

### 3.1 双层导航（左侧）

- **顶层 = 架构骨架**（体现技术先进性，服务汇报叙事）：
  - `Wiki 大脑` · `本体建模 DORA` · `交付 Claw`
- **二层 = 业务入口**（点进去落到实际干活的地方），例如：
  - `交付 Claw` 下：`预案生成` · `计划排期` · `现场工勘` …
  - `计划排期` 内再落到：`人货站态势` / `关键路径` / `沙箱推演` …
- 导航按角色权限差异化展示（不同角色可见模块不同）。

### 3.2 右侧常驻对话栏（Claw）

- Claw 对话**常驻右侧边栏**，始终可对**当前页面上下文**提问 / 下指令（Copilot 范式）。
- 对话能引用当前页的人货站/排期对象，并能触发动作（如发起一次沙箱推演）。

### 3.3 固定下钻层级

```
项目群 → 项目 → 机房 → PoD → 活动
```

每层只展示该层的关键指标与异常，下钻进入下一层。

## 4. 人货站呈现要求

人货站不是三个静态展示维度，而是**约束链**："人等货，货等站"。前端必须表达出这种**传导关系**与**就绪/阻塞**状态，而不仅是数字罗列。

- **站**（客户机房/入场条件）：多少机房、每机房放几个 PoD、是否具备入场条件。
- **货**（PoD/设备）：多少 PoD、是否到位、是否满足现场实施条件（齐套）。
- **人**（施工/调试队伍）：多少队伍、每队多少人、是否就位、是否满足当前阶段需要。

呈现建议：用"站→货→人"的就绪链路视觉化每个 PoD 的阻塞点（如"站 ready 但货未齐套"），让传导关系一眼可见。

---

## 5. 角色视角：一套数据，多个镜头

顶部提供视角切换器（全局视角 / PD / TD），同一份数据切换关注点（领导不直接操作系统，通过「全局视角」查看与汇报）：

- **全局视角**：全局态势、整体进度、关键风险、是否能按合同交付。最少细节、最强结论；供领导汇报查看。
- **PD 镜头**：履约统筹——合同节点、入场/到货/资源就位、延期影响、需协调事项。
- **TD 镜头**：技术交付——PoD 安装条件、上电/点亮/调试/测试依赖、技术风险与返工影响。

## 6. 主页方案：出两版，实做后选一版

两版都交给 Claude Design 实现，后续按效果择优。

### V1 · 交付态势大屏（作战指挥中心气质）

- 进入即是"交付盘子"，**不以空对话框作为入口**。
- 第一屏建议模块：
  - **人货站全局健康度**：项目群/项目维度的站、货、人就绪概览（可用环形/链路/热力表达）。
  - **AI 关键研判**：系统主动给出的几条结论（如"X 项目机房延期将导致移交里程碑顺延 N 天"），每条可下钻看依据。
  - **风险预警**：按严重度排序的阻塞与冲突，区分"不可满足"与"可满足但有风险"。
  - **关键里程碑/合同交付达成态势**。
- 右侧常驻 Claw 对话；对话作为辅助，不抢主屏。
- 目标体感：作战指挥中心，系统在主动盯盘。

### V2 · 导航 + 对话引导（Agentic 入口气质，源自现有草图）

- 左侧双层导航（见 3.1）。
- 中央 **AIDA 系统图标** + 下方对话框，主页引导用户**上传项目上下文**（HLD、BOQ、合同、到货表、机房信息等），由系统解析并建立项目。
- 目标体感：从"对话+喂资料"开始，系统据此自动构建交付盘子，凸显 Agentic 能力。

## 7. AI 可解释 / 智能化出口（建议方案，待确认）

> 这是把底层四层架构"翻译"给用户的关键，建议作为一等公民。以下为建议，待你确认细化。

- **研判可溯源**：每条 AI 研判/风险可展开看「依据（引用 Wiki 大脑/本体）+ 推理（关键路径、约束传导）」。
- **排期 should-be vs as-is 对照**：任何排期调整都展示"原计划基线 vs 调整后"的对照视图，并标注变化原因、受影响链路、是否仍满足合同。
- **沙箱推演可见**：调整先在沙箱执行，UI 明确区分"沙箱推演结果（未写回）"与"正式计划"，确认后才写回。
- **关键路径高亮**：在排期/活动视图中高亮关键路径，节点变更时提示影响范围。

## 8. 可配置项（建议方案，待确认）

部分信息应支持手动配置而非写死，建议至少包括：

- 人货站基础信息（队伍数/人数、PoD 数与归属、机房数与承载）。
- 锚点与优先级、工期规则、风险规则、调整策略偏好（呼应业务侧"配置化原则"）。
- 角色可见模块与关注指标。

> 待你确认：哪些由用户手动配置，哪些由系统自动计算/推理。

---

## 9. 视觉系统（气质护栏，不锁死配色）

> 具体配色**不预先定死**。请 Claude Design 在以下气质护栏内**多生成几版配色方案**（浅色/深色、冷调/暖调均可探索），再对比择优。曾尝试的"深炭底 + 大面积红"偏脏，应避免高饱和红大面积铺在深色底上。

Logo 与品牌字标以 `aida-delivery` 知识库的 `03_品牌与Logo/AIDA_Logo设计方案.md` 为准（AIDA Gateway：大写 A 架构之门）。

**气质护栏（必须满足，几版都不能跑偏）**：

- 企业级、冷静、精确、有结构感、可信，能直接用于面向顶层领导的汇报。
- 层级靠海拔/对比/边框/克制的微动效表达，而非扁平平铺。
- **禁廉价特效**，但允许有纪律的深度与动效。
- 红 = "要紧事"语义（品牌重点/风险），保持克制，避免大面积铺红。
- 禁用：机器人/脑/电路板/盾牌/地球网格等泛 AI 图形、霓虹、3D、蓝紫大面积渐变、消费品风。

**可参考方向（非强制）**：Palantir Gotham/Foundry 的指挥中心气质——深色、高对比、网格密集、近单色、技术化字体；以及红黑灰的企业汇报底色。仅作灵感，不作硬性配色要求。
