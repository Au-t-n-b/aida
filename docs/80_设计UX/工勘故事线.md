# 智慧工勘 · 用户故事线（前端优化基准）

> **文档定位**：这是 zhgk 前端/SDUI 优化的**叙事基准与对照真相**。
> 它把「目前 zhgk SDUI 实现」与「业务专家原型 `smart_survey_v8`（同事手写 HTML）」放到同一条用户旅程上对照，
> 标出每一幕的**差距热区**与**优化方向**。
> 具体的优化任务排期以 [ROADMAP.md](../../ROADMAP.md) 为唯一真相，本文件只负责「为什么这么改、改成什么体验」。
> SDUI 投影规范见 [SDUI.md](../30_skill开发/31_手写规范/SDUI.md)；视觉 token 见 [DESIGN.md](DESIGN.md)。

---

## 0 · 两种范式的根本差异

| | 业务专家原型 `smart_survey_v8` | 当前 zhgk |
|---|---|---|
| 范式 | **双态导航 + 多页表格作业台**（前端自持状态，用户主动 CRUD） | **SDUI 单页投影 + 对话驱动 + HITL 门控**（后端投影界面树，前端零改） |
| 世界中心 | **机房**（"我在管几个机房"） | **意图**（"你要干哪种活") |
| 用户驱动力 | 直接操作表格（编辑/筛选/批量/下钻） | 左侧 Claw 对话 + HITL 选择 + 文件上传 |
| 优势 | 作业态交互丰富、信息密集 | 全流程闭环完整、AI 评估体系化、数据落点规范 |

> **优化总原则**：借鉴原型的「丰富作业态体验」，但**不破坏 SDUI「后端投影、前端零改」的架构**——
> 凡需要更细的交互，优先通过 metrics 扩充 + 既有 SDUI 节点组合实现，用户操作通过 PostUserMessage 回投给 Claw，由后端改 state。

## 1 · 主角与场景

**李泽**，华东液冷智算中心 Phase-1 入场勘测现场总负责人。
贯穿全程的常驻角色：左侧 **Claw 勘测助手**——它是李泽推进每一步的「遥控器」，不是装饰。

下面以**全流程工勘（survey_work）**为主线（最完整），其余三种意图作为岔路标注。

## 2 · 八幕故事线

### 第 0 幕：落地 —「我现在在哪，整体怎么样了」
李泽打开平台，第一眼想看的是**全局态势**，不是某张表。

- **原型**：落地即 `overview` 态——顶部勘测窗口时间条（计划 vs 实际双轨 + 超时红色预警），下面 3D 机房俯视总览（三机房卡片 + 四值统计 + 进度 + 入口）。一屏看完全局。
- **现状**：`idle` 态只有一张文字引导卡，启动后才出现 3D 总览，且总览与作业仪表盘**同屏堆叠**。
- **优化**：① 3D 机房总览**提升为落地首屏**（未启动时它就是态势入口）；② idle 引导卡缩为总览里的空状态浮层。
- **灵魂**：原型以「机房」为中心、符合李泽心智；我们现在先问「你要干嘛」——**节奏反了**，应先见机房再选动作。

### 第 1 幕：选择切入点 —「我要对这个机房做什么」
- **原型**：每张机房卡三入口（勘测主线 / 通液电子流 / 液冷湿材质 AI 审核）。
- **现状**：真实机房卡已做成**四意图启动盘**，点击投递 `/intent <value>`——**这点我们比原型更对**。
- **优化**：区分入口视觉权重，survey_work 为主入口（大、primary），其余次级。
- **岔路**：`scene_suggest`/`supplement` 为短岔路（选完即出结果结束）；`report_gen` 为快进路（已有数据直接出报告）；**仅 survey_work 走完整剧情**。

### 第 2 幕：进入作业台 —「确认这是我要勘的那个机房」
- **原型**：`RoomContextBar` 信息密集——机房名 + 编号 + 勘测窗口 + 负责人姓名电话 + 工具组 + 范围 + 切换机房。
- **现状**：上下文条只有一行 `📍 机房名 · 意图 · 项目编号` + 返回按钮，负责人/窗口/范围全缺。
- **优化**：从 `project_info.json` 读 `surveyor / survey_date / sub_scenes`，补成两行密集信息。**改动最小、增益最大。**两态导航（`/overview` 返回）已对齐，守住别再同屏堆叠。

### 第 3 幕：方案识别 —「这是什么代际、什么制冷，该勘哪些条目」
- **原型**：被前置假设掉（demo 直接从条目编制开始）。
- **现状**：`determine_gen` + `filter_build`，KPI 出「代际制冷/条目数/场景」，中列出**条目预览表**（前 10 条），识别不出时 HITL 选择。**我们独有、做得好。**
- **优化**：条目预览表借原型 Phase1 的**列筛选下拉**（按分类/来源多选），确认前可快速翻检全量。

### 第 4 幕：条目编制与确认 —「这份清单我认可，可以开干」
- **原型**：**Phase1 是完整可编辑表格**——搜索、列头下拉多选筛选、行内双击编辑、勾选需要、添加、导出、分页。用户是表的主人。
- **现状**：`confirm_table` 是 HITL ChoiceCard（确认 / 重建），**没有逐条微调**。
- **优化**：**范式差距最大的一幕**。不照搬可编辑大表（违背 SDUI 原则），但提供**轻量编辑入口**（剔除某条/改反馈方式），通过 PostUserMessage 投给 Claw 改 state。让"确认"从二选一变成"可微调后确认"。

### 第 5 幕：任务下发与数据反馈 —「派下去，等数据填回来」
- **原型**：**Phase2 亮点是轮次多选对比**——R1~R5 可同时勾选，表格横向扩展多列实勘值、逐列着色；行点击详情抽屉含「逐轮实勘值」对比表。
- **现状**：`task_dispatch`（GKCLAW 邮件下发 HITL）+ `wait_survey`（FilePicker 上传，支持多轮）。多轮历史只有**静态二维表**（轮次/填写数/填写率/较上轮），无横向逐轮对比、无下钻。
- **优化**：`wait_survey` 把每轮「满足/不满足/无法识别」存进 `survey_round_history`，复勘历史表展示**每轮评估趋势**。GKCLAW 任务卡是我们的优势，保留。

### 第 6 幕：AI 五值评估 —「机器先帮我判一遍」
- **原型**：评估体现在 Phase2「AI 评估」列 + 轮次四值快照。
- **现状**：`assess` → **五值评估面板**（BarChart + 满足率 + 三类告警 Alert）。**比原型更体系化。**
- **优化**：面板「未勘测/无法识别」做成**可点击下钻**到对应条目。

### 第 7 幕：问题闭环 —「哪些不达标，怎么跟进整改」
**原型着墨最重、我们差距最明显的一幕。**

- **原型**：**Phase3 是 demo 重点**——问题列表（来源/严重度/责任人/状态/SLA）+ 状态分段筛选（带计数）+ 批量勾选下发复勘 + 重型详情抽屉（标准vs实勘 / AI 建议 / 审批 / 责任人 / 生成-审批-转派时间 / **活动时间线** / 底部「转专家复核·下发复勘·关闭问题」）。KPI 含「问题18·已关闭11·关闭率61%·遗留7(高3中3低1)」分级。
- **现状**：`issue_list` → 问题清单表（序号/描述/状态/建议）+ 详情抽屉（**只深挖第 1 条**）。`resurvey_gate` 提供复勘/跳过 HITL。**缺：分级统计、状态筛选、批量、责任人、SLA、活动时间线、逐条操作。**
- **优化**：① `issue_list.py` 补 `issue_high/mid/low` + 关闭率，KPI 升级为「N 条·高h中m低l·关闭率x%」；② 抽屉从「只看第 1 条」扩展为**清单内任意条目下钻**；③ 借鉴**活动时间线**（按问题维度聚合已有日志）。

### 第 8 幕：报告生成与审批分发 —「出报告，专家签字，发出去」
- **原型**：止于问题闭环，无独立报告/分发阶段。
- **现状**：`report_gen_run`（Word 报告 + 风险识别表）+ `report_distribute`（HITL 审批：通过/驳回/暂存 → 邮件分发四件套）+ **风险告警 RiskList** + **审批流程卡**。**整幕我们独有、最完整。**
- **优化**：审批卡借原型抽屉的**底部固定操作栏**视觉规范，让「通过/驳回/暂存」更像决策动作。

### 贯穿全程：左侧 Claw 对话 —「我的遥控器」
- **原型**：常驻可折叠对话面板，对话流**内嵌 TodoList 卡片**（点阶段跳转），底部**业务化快捷 chip**（SLA 风险/起草复勘/R3 轮总结/呼叫专家），对话与作业区双向联动。
- **现状**：左侧会话 + 右侧 SDUI 投影，3D 入口/上下文条已会投递 PostUserMessage。但待办在右栏（非对话流），chip 通用。
- **优化**：① 右栏阶段待办**可点击跳转**（投递指令让 Claw 续跑/聚焦该步）；② 快捷 chip **业务化 + 上下文感知**。

```
        原型 smart_survey_v8              当前 zhgk
        ──────────────                   ──────────────
```

## 4 · 落地批次（按「故事幕」而非技术模块推进）

> 每改完一批，李泽在一个**完整环节**上的体验就变好。具体排期/拆解进 [ROADMAP.md](../../ROADMAP.md)。

1. **第一批 · 落地体验**：第 0 幕总览提为首屏 + 第 2 幕上下文条补全 → 「进门」体验立刻像成品。
2. **第二批 · 问题闭环**：第 7 幕分级统计 + 任意条目下钻 + 活动时间线 → 补最大差距。
3. **第三批 · 多轮演进**：第 5 幕逐轮趋势 + 第 6 幕评估下钻 → 把「多轮勘测」讲清楚。
4. **第四批 · 精修**：第 4 幕轻量编辑 + 全程对话联动 + chip 业务化。
