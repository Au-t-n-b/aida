## 总体架构

```text
前端 /proposal
  └─ ProposalDataProvider（useProposalData）
       ├─ proposal-data-service.ts  加载/缓存/筛选/保存
       └─ xlsx-io.ts                调用 Mock API

Agent Mock API（uvicorn :7401，Vite 代理 /api/v1）
  ├─ GET  /api/v1/mock/file?logicalPath=   读 xlsx/docx/md
  ├─ POST /api/v1/mock/file                写输出 xlsx
  ├─ POST /api/v1/proposal/parse/tech-proposal   上传解析技术建议书
  └─ POST /api/v1/proposal/parse/testcases       上传解析测试用例

多模态解析（officecli 主路径，失败降级 openpyxl/python-docx）
  └─ skills/officecli/  +  agent/services/officecli_*.py
```

**本地启动**：

```bash
# Agent（必须先启动，否则保存/解析会失败）
cd aida && python -m uvicorn agent.main:app --host 127.0.0.1 --port 7401 --reload

# 前端
cd aida/frontend && npm run dev
```

## 1. 责任矩阵信息（第 9 章）

![责任矩阵 UI](image/0610需求/1781077294058.png)

### 功能需求

- **加载责任矩阵**  
  - 首次进入「交付预案」：若该项目无已保存责任矩阵，从 `组织资产/责任矩阵标准模板.xlsx` 加载模板填充表格（接口未就绪，走 Mock API）。  
  - 若已有保存版本：从 `早期介入/交付预案/输出结果/项目责任矩阵.xlsx` 加载**最新版本**。  
  - 若用户修改后**未保存**：再次进入仍保留编辑内容（`sessionStorage` 草稿，优先级高于已保存文件）。

- **本地存储责任矩阵**  
  - 路径：`{项目根}/早期介入/交付预案/输出结果/项目责任矩阵.xlsx`  
  - 触发：右上角「保存草稿」或「生成预案并决策」  
  - 字段：项目名称、技术栈、活动分类、活动、GTS、华为云、伙伴、客户、版本号  
  - 版本号：从 1 递增；**保存草稿**时版本不变，**生成预案并决策**时 +1

### 实现说明

| 项 | 说明 |
|:---|:---|
| 加载优先级 | `sessionStorage` 草稿 > 已保存 xlsx > 组织资产模板 > 内置静态兜底 |
| 输出 xlsx 格式 | 第 1 列「项目名称」、末列「版本号」，**每行数据均填充相同项目名称与版本号**（单行表头，无 meta 行） |
| 前端组件 | `chapters/raci-chapter.tsx`（可编辑下拉） |
| 路径常量 | `frontend/src/data/project-paths.ts` → `ProposalMockPaths` |
| 数据服务 | `frontend/src/lib/proposal-data-service.ts` |
| 解析/落盘 | `agent/services/proposal_parse.py` |

## 3. 验收策略（第 11 章）

### 功能需求

- **展示验收策略表**  
  - 首次进入：从 `早期介入/交付预案/输入文件/技术建议书.docx` 多模态解析填充，**只读**；首次加载后 `sessionStorage` 缓存。  
  - 无上传文件/无解析结果时：展示表头 + 一行空数据。  
  - 表头：分类、验收方案、验收标准、验收里程碑、验收文档、回款条款、回款里程碑。

- **生成验收策略表（上传触发）**  
  ![文件上传入口](image/0610需求/1781074526853.png)  
  - ClawRail 文件上传区新增「**技术建议书**」条目（`.docx/.pdf`）。  
  - 上传 `技术建议书.docx` 后调用多模态解析，提取要素刷新第 11 章验收策略表。

- **多模态解析（officecli）**  
  - Skill 落盘：`skills/officecli/SKILL.md`（内网 SSOT 可从此处同步更新）  
  - 管线 A：`scripts/service_advice_post.py` — 5 类验收硬编码模板 + 关键词全文检索 + docx 表格直提  
  - 引擎：`skills/officecli/reference/officecli-win-x64.exe`（Windows）  
  - 降级：officecli 不可用时回退 `python-docx`（`proposal_parse.py`）

- **本地存储验收策略**  
  - 路径：`早期介入/交付预案/输出结果/验收策略.xlsx`  
  - 触发：「保存草稿」/「生成预案并决策」  
  - 字段：项目名称、分类、验收方案、验收标准、验收里程碑、验收文档、回款条款、回款里程碑、版本号  
  - 输出格式：同责任矩阵（项目名称首列、版本号末列、每行同版本）

### 实现说明

| 项 | 说明 |
|:---|:---|
| 上传解析 API | `POST /api/v1/proposal/parse/tech-proposal` |
| 首次加载 API | `GET /api/v1/mock/file?logicalPath=...技术建议书.docx` |
| 空态规则 | 无上传文件 / 无解析结果 → 表头 + 一行空数据（**不回退**静态 mock） |
| 前端事件 | 上传成功后 `aida:proposal-acceptance-parsed` → `useProposalData` 刷新 |
| 关键文件 | `agent/services/officecli_parse.py`、`skills/officecli/scripts/service_advice_post.py`、`claw-rail.tsx` |

## 4. 测试用例（第 12 章）

### 功能需求

- **加载时机（上传门控）**  
  - **首次进入且未上传「场景测试用例」**：第 12 章展示空状态「暂无测试用例」。  
  - **上传「场景测试用例」后**：触发加载与展示（同会话内 `sessionStorage` 缓存，刷新页面仍保留）。

- **生成/加载流程（上传触发）**  
  1. ClawRail 文件上传区「**场景测试用例**」（`.docx/.pdf`）。  
  2. 上传后调用 `POST /api/v1/proposal/parse/testcases`。  
  3. 后端**始终**从 `早期介入/交付预案/输入文件/测试用例new.xlsx` 读取默认条目（目录 SSOT）。  
  4. 若上传为 `.docx`：officecli + `testcase_post` 解析，按 `用例编号/case_id` 与 xlsx 合并（docx 优先补全 steps/expects 等详情）。  
  5. 若上传为 `.pdf` 或 docx 解析失败：仅返回 xlsx 默认条目。  
  6. 前端 `filterTestCases()` 按卡规模筛选后展示，默认**全部勾选**。  
  7. 解析失败、无内容或未上传：展示空状态「暂无测试用例」。

- **多模态解析（officecli）**  
  - xlsx 默认条目：`officecli get` 读表（`officecli_client.xlsx_sheet_rows`）  
  - docx 上传：`scripts/testcase_post.py` 结构化提取（Heading3 + 表格字段）  
  - mock 生成脚本：`frontend/scripts/gen-mock-testcases-xlsx.mjs`（从 `proposal-testcases.ts` 导出 54 条）

- **测试用例筛选**  
  - 从 `早期介入/合同/输出结果/建模仿真设备信息表.md` 解析卡规模（如 `液冷384卡` → 384）  
  - 对「集群集合通信测试」「模型集群训练性能测试」类用例：从用例名提取节点数，卡数 = 节点数 × 8，**仅保留卡数 ≤ 项目卡规模**的用例  
  - 其余分类（计算子系统、灵衢子系统等）全部保留

- **本地存储测试用例**  
  - 路径：`早期介入/交付预案/输出结果/测试用例.xlsx`  
  - 触发：「保存草稿」/「生成预案并决策」  
  - 内容：勾选标记 + 用例完整字段（含步骤、预期结果等）

### 实现说明

| 项 | 说明 |
|:---|:---|
| 上传门控 | `sessionStorage` 键 `aida:proposal:tc-uploaded`；未标记时 `loadTestCasesIfReady()` 返回 `[]` |
| 上传后缓存 | `aida:proposal:tc-cache` 存筛选后用例；`applyTestCasesFromUpload()` 写入 |
| 上传解析 API | `POST /api/v1/proposal/parse/testcases`（xlsx 默认 + docx 合并，响应含 `source` 调试信息） |
| 前端事件 | 上传成功后 `aida:proposal-testcases-parsed` → `useProposalData` 刷新 |
| ClawRail 条目 | `scenarioTc` ·「场景测试用例」`.docx/.pdf` |
| 筛选逻辑 | `proposal-data-service.ts` → `filterTestCases()` |
| 前端组件 | `chapters/test-case-chapter.tsx`（树形勾选 + 详情） |
| 后端合并 | `officecli_parse.merge_testcases_with_upload()` |
